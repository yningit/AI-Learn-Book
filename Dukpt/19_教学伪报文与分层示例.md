# 教学伪报文与分层示例

## 1. 为什么要做“伪报文”示例

很多人理解了概念之后，真正一看到终端上送字段还是会发懵。原因是：

- 真实报文往往又长又杂。
- 厂商格式差异很大。
- 一开始就看真实生产报文，反而不利于建立清晰模型。

所以这一篇不用真实厂商格式，而是用“教学化伪报文”帮助你看懂：

- 哪些字段属于业务元数据。
- 哪些字段属于密钥/路由辅助信息。
- 哪些字段属于密文对象。
- 哪些字段可能受 MAC 覆盖。

## 2. 一个最小化终端交易伪报文

下面先看一个极简教学版示例：

```text
MessageType: SALE_REQUEST
MerchantId: M123456
StoreId: S0008
TerminalId: T900188
OrderId: O202603180001
Amount: 12800
Currency: CNY
Timestamp: 2026-03-18T10:15:20
TransactionMode: CHIP_OR_SWIPE
KSN: FFFF9876543210E00015
EncryptedDataBlock: <ciphertext>
PinBlock: <encrypted-pin-block>
Mac: <message-authentication-code>
```

这个示例不代表某个真实厂商格式，但足够帮助你建立阅读顺序。

## 3. 先把字段分成四层来看

### 第一层：业务元数据

例如：

- MessageType
- MerchantId
- StoreId
- TerminalId
- OrderId
- Amount
- Currency
- Timestamp

这些字段主要驱动：

- 交易编排
- 订单关联
- 路由决策
- 审计和对账

### 第二层：密钥与解密辅助信息

例如：

- KSN

它帮助系统：

- 找到正确派生路径
- 找到正确密钥域
- 校验终端身份上下文

### 第三层：敏感对象密文

例如：

- EncryptedDataBlock
- PinBlock

它们回答的是：

- 敏感对象没有以明文形式在中间系统出现。

### 第四层：完整性与认证对象

例如：

- Mac

它回答的是：

- 这条报文有没有被改过。

## 4. 为什么 `EncryptedDataBlock` 和 `PinBlock` 不能混为一谈

表面上看，它们都像一段密文。但它们在系统里的语义完全不同：

- `EncryptedDataBlock` 更偏 P2PE / Data Encryption 语境。
- `PinBlock` 更偏 PIN 域语境。

这意味着：

- 用途键可能不同。
- HSM 处理路径可能不同。
- 合规要求可能不同。

所以在看报文时，不能因为两者都长得像“密文块”就把它们当成同一类对象。

## 5. 为什么 `Mac` 又不能和前两者混为一谈

`Mac` 通常不是“另一段加密数据”，而是：

- 对报文关键部分做出的消息认证结果。

因此它的价值不是“把内容藏起来”，而是：

- 证明内容未被篡改。

## 6. 一个更完整的教学分层示例

下面给出一个更接近系统设计的教学分层：

```text
Transport Header
  ProtocolVersion: 1.0
  MessageType: SALE_REQUEST

Business Context
  MerchantId: M123456
  StoreId: S0008
  TerminalId: T900188
  OrderId: O202603180001
  Amount: 12800
  Currency: CNY
  Timestamp: 2026-03-18T10:15:20

Security Context
  KSN: FFFF9876543210E00015
  KeyMode: DUKPT_DATA_PIN_MAC
  SecurityProfile: PROFILE_A

Protected Objects
  EncryptedDataBlock: <ciphertext>
  PinBlock: <encrypted-pin-block>

Integrity
  Mac: <mac>
```

这个结构的学习意义在于：

- 你可以一眼看出哪些字段是为了业务。
- 哪些字段是为了安全路由。
- 哪些字段是敏感密文。
- 哪些字段是完整性保护结果。

## 7. 主机侧如何读这类报文

主机通常不会“从头到尾平等地处理所有字段”，而是会按层处理。

一个典型顺序可能是：

1. 先根据头部识别消息类型。
2. 再根据业务上下文做基础路由。
3. 再根据 KSN 和安全上下文选择密钥域。
4. 再校验 MAC 或其他完整性控制。
5. 再分别进入 Data 解密链路和 PIN 域链路。

这里最关键的一点是：

- 主机看到报文，不等于所有字段都进入同一个处理函数。

## 8. 为什么真实系统里“字段边界”比“字段名字”更重要

真实项目里，字段名字可能千差万别，但你只要抓住边界，就能快速判断它属于哪一层：

- 它驱动业务吗。
- 它帮助密钥路由吗。
- 它本身是密文载荷吗。
- 它是完整性校验对象吗。

这比死记某个字段名更重要。

## 9. 一个管理报文伪示例

支付终端不只发交易报文，还可能发管理或参数相关报文。

例如：

```text
MessageType: PARAM_DOWNLOAD_REQUEST
TerminalId: T900188
Timestamp: 2026-03-18T10:20:00
ProfileVersion: P-202603
Mac: <message-authentication-code>
```

这个报文里可能没有卡数据，但 `Mac` 仍然非常重要，因为：

- 管理报文最怕被伪造或篡改。

这也说明：

- MAC 的重要性并不依赖于“有没有卡数据”。

## 10. 一个失败交易伪示例

```text
MessageType: SALE_RESPONSE
OrderId: O202603180001
TerminalId: T900188
ResponseCode: 912
ResponseMessage: HOST_SECURITY_CHECK_FAILED
ReferenceId: R9988123
Mac: <message-authentication-code>
```

这个例子想说明两件事：

- 失败响应也常需要完整性保护。
- 排障时最有价值的通常是 `OrderId / TerminalId / ResponseCode / ReferenceId`，而不是把整条敏感原报文全量打到日志里。

## 11. 你应该怎样使用这篇示例

这篇不是替代厂商协议文档，而是帮助你建立阅读真实报文的框架。一个建议的使用方式是：

1. 先看这一篇，学会按层读报文。
2. 再去看真实厂商格式，给每个字段贴上“业务 / 路由 / 密文 / 完整性”标签。
3. 再回到 [14_厂商命名与版本语境提示.md](./14_厂商命名与版本语境提示.md)，把厂商命名放到正确层次上理解。

## 12. 一句总结

教学伪报文的价值不在于模拟真实厂商格式，而在于训练你按“业务上下文、密钥上下文、密文载荷、完整性结果”四层去阅读终端到主机的安全报文。只要这个分层习惯建立起来，之后看任何厂商协议都会快很多。
