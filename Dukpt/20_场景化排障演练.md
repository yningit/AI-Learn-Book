# 场景化排障演练

## 1. 这一篇的目标

前面的文档已经讲了很多概念、机制和结构。这一篇不再新增大量新概念，而是把已有内容放进几个典型故障现场中，让你练习：

- 先看什么。
- 哪些问题优先排除。
- 如何把 DUKPT、用途键、HSM、注入、MAC、PIN 域这些知识真正用起来。

## 2. 演练方法

每个场景都按下面四步来看：

1. 现象是什么。
2. 最可能涉及哪一层。
3. 优先检查什么。
4. 容易误判成什么。

## 3. 场景一：只有一台终端突然开始解不开

### 现象

- 某台终端之前正常。
- 今天开始连续报解密失败。
- 同门店其他终端正常。

### 最可能涉及哪一层

- DUKPT 状态层
- 设备生命周期层
- 台账层

### 优先检查什么

1. 这台终端最近是否升级、返修、重启或替换。
2. 当前 KSN 是否异常回退、重复或跳变。
3. 设备是否仍绑定在正确环境和密钥域。
4. 最近是否发生过重新注入或参数切换。

### 容易误判成什么

- 容易一上来怀疑 HSM 算法错了。

但如果只有一台设备异常，通常先看状态和台账，比先看算法更高效。

## 4. 场景二：一批设备上线后全部失败

### 现象

- 某批新设备上线后全部失败。
- 老设备继续正常。

### 最可能涉及哪一层

- 注入层
- 批次台账层
- 环境隔离层

### 优先检查什么

1. 这批设备是否使用了错误环境的注入材料。
2. 批次与 KSN / IPEK / 域映射是否一致。
3. 终端型号、固件版本是否与主机预期兼容。
4. 主机是否为这批设备配置了正确路由和算法模式。

### 容易误判成什么

- 容易误判为“主机整体故障”。

但如果老设备正常，新设备整批异常，批次问题通常优先级更高。

## 5. 场景三：PIN 相关交易失败，但不输 PIN 的交易正常

### 现象

- 刷卡或挥卡交易正常。
- 一旦涉及 PIN 输入，就失败。

### 最可能涉及哪一层

- PIN 域
- PIN Key
- HSM PIN 处理路径

### 优先检查什么

1. 终端 PIN 输入链路是否正常。
2. PIN Key / PIN 域配置是否正确。
3. 主机是否把 PIN Block 路由到了正确 HSM PIN 功能。
4. 是否误把 PIN 对象当成普通 Data 对象处理。

### 容易误判成什么

- 容易误判为“整条交易链路有问题”。

但“有 PIN 才失败”通常在强烈提示：

- Data 域和 PIN 域的路径不一样。

## 6. 场景四：MAC 校验失败，但密文解密逻辑看起来没问题

### 现象

- 主机报告完整性校验失败。
- KSN 看起来正常。
- 终端确实有上送密文载荷。

### 最可能涉及哪一层

- MAC 用途键层
- 报文封装层
- 完整性覆盖字段定义层

### 优先检查什么

1. MAC 覆盖的字段集合双方是否一致。
2. 报文某些字段是否在中间层被改写。
3. 是否选错 MAC Key 或 MAC 用途变体。
4. 是否存在不同厂商模式或不同版本格式混用。

### 容易误判成什么

- 容易把它误判为“数据密钥错误”。

但 MAC 失败往往首先是完整性和报文结构问题，不一定是 Data 解密问题。

## 7. 场景五：测试环境正常，生产环境失败

### 现象

- 同一套流程测试环境通过。
- 生产环境失败。

### 最可能涉及哪一层

- 环境隔离层
- HSM 域层
- 密钥域与路由层

### 优先检查什么

1. 生产和测试是否使用了不同 BDK / PIN / MAC 域。
2. 终端是否误拿了测试注入材料。
3. 主机是否在生产仍指向测试路由或旧配置。
4. 是否有厂商模式在不同环境中配置不一致。

### 容易误判成什么

- 容易认为“代码没问题，应该是生产网络不稳定”。

但在支付安全场景里，环境和密钥域错位通常比网络抖动更值得优先排查。

## 8. 场景六：参数下载失败或终端远程管理异常

### 现象

- 交易可能还正常。
- 但参数下载、远程命令、管理同步失败。

### 最可能涉及哪一层

- MAC / 管理报文完整性层
- 终端管理用途键层
- 厂商格式层

### 优先检查什么

1. 管理报文是否使用了与交易报文不同的认证路径。
2. 管理用 MAC / Authentication Key 是否配置正确。
3. 终端和主机对参数版本、profile、格式是否一致。

### 容易误判成什么

- 容易误判为“只要交易能走，安全配置一定没问题”。

但管理链路常常有独立安全控制。

## 9. 场景七：返修设备重新上线后偶发失败

### 现象

- 设备返修后重新上线。
- 大多数交易正常，但偶发失败。

### 最可能涉及哪一层

- 生命周期层
- 注入层
- 状态迁移层

### 优先检查什么

1. 返修后是否重新注入或重新绑定。
2. 设备身份与原台账是否 still 一致。
3. 固件升级是否影响 KSN / 状态持久化。
4. 是否发生了用途键或管理参数的不完整恢复。

### 容易误判成什么

- 容易认为“偶发就先不管”。

但在支付场景里，偶发通常意味着状态或边界没有完全闭合。

## 10. 一个通用的排障顺序模板

如果你不确定从哪里下手，可以先按下面顺序排：

1. 先判断是单设备、单批次、单环境，还是全局问题。
2. 再判断是 Data 域、PIN 域、MAC 域，还是管理链路问题。
3. 再看设备状态、台账、返修、升级、注入记录。
4. 再看主机密钥域、HSM 路由、用途键和格式模式。
5. 最后才深入具体算法和实现细节。

这能显著减少“上来就研究密码学细节，却忽略台账和环境”的低效排障。

## 12. 一句总结

真正高效的排障，不是记住更多术语，而是学会先把问题归到正确层次：设备、状态、域、用途、报文、完整性、注入还是环境。只要层次归对了，很多复杂问题都会迅速变简单。
