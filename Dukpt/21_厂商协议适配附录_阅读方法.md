# 厂商协议适配附录：阅读方法

## 1. 这一附录解决什么问题

当前文档集已经把：

- P2PE
- DUKPT
- PIN / Data / MAC
- HSM
- 注入
- Key Block

这些核心概念讲清楚了。

但一旦你真正拿到某家终端厂商的资料，新的困难就会出现：

- 字段名完全不一样。
- 有很多厂商自定义缩写。
- 报文结构和我们教学模型不一样。
- 资料里会把“算法、封装、用途、产品模式”混着写。

这一附录的目标，就是教你如何把陌生厂商协议重新翻译回我们已经建立好的知识框架。

## 2. 阅读厂商协议时最容易犯的错

### 2.1 直接按字段名理解字段

这是最常见的错误。因为同一个含义，不同厂商可能叫：

- `KSN`
- `KeySN`
- `Key Serial`
- `Derivation Counter`
- `Security Serial`

如果你只盯着名字，很容易误判。

### 2.2 把产品名当标准名

例如某资料说：

- `OnGuard Mode`
- `Secure Data Packet`
- `Host Encrypted Format`

这些名字可能是厂商产品层命名，不等于行业标准名词。

### 2.3 看到密文块就默认它一定是卡数据块

实际上它可能是：

- PIN Block
- Data Block
- 管理报文中的认证块
- 参数下载保护块

### 2.4 只看终端侧，不看主机侧输入要求

厂商协议适配的重点从来不是“终端发了什么”这么简单，而是：

- 主机要如何识别。
- HSM 要走哪条处理路径。
- 哪些字段进入路由，哪些字段进入解密，哪些字段进入 MAC 校验。

## 3. 一套推荐的阅读顺序

当你拿到厂商协议时，建议按下面顺序看，而不是从头顺着往下读。

### 第一步：先找安全模式总览

优先找到：

- 终端支持哪些安全模式。
- 哪些模式与 DUKPT 相关。
- 哪些模式与 PIN / Data / MAC 相关。
- 是否有按年份、版本或 profile 区分的模式。

这一阶段不要急着看细节，先画模式地图。

### 第二步：找报文层结构

先搞清楚：

- 一条请求报文由哪几段组成。
- 哪些段是业务字段。
- 哪些段是安全字段。
- 哪些段是密文或认证结果。

### 第三步：找密钥与用途说明

优先识别：

- 是否使用 DUKPT。
- 是否区分 PIN / Data / MAC。
- 是否使用 variant key 或 usage attribute。
- 是否存在厂商定义的用途模式。

### 第四步：找主机侧处理说明

你必须能回答：

- 主机如何定位 KSN。
- 主机如何知道这是哪个模式。
- 主机先校验 MAC 还是先解密。
- PIN Block 是否走独立 HSM 路径。

### 第五步：找兼容与版本说明

要特别注意：

- 旧型号和新型号是否字段不同。
- 同一模式是否有多个 profile。
- 某些厂商是否同时支持传统模式和新模式。

## 4. 阅读时建议先做一张“模式地图”

你可以为每个厂商模式建立一张最小表：

```text
模式名称:
厂商/型号:
属于哪一层: 算法 / 用途 / 报文 / 产品模式
是否使用 DUKPT:
是否有 KSN:
是否包含 PIN:
是否包含 Data Encryption:
是否包含 MAC:
主机如何识别:
HSM 处理路径:
兼容版本:
```

这张表的价值非常高，因为它能强迫你把模糊命名拆成结构化信息。

## 5. 厂商协议里最关键的 8 类字段

不管字段名怎么变化，你都应该优先识别下面几类：

1. 交易类型字段。
2. 终端身份字段。
3. 商户与路由字段。
4. KSN 或密钥定位字段。
5. Data Encryption 相关密文字段。
6. PIN Block 相关字段。
7. MAC / 完整性相关字段。
8. 模式 / profile / 安全版本字段。

只要这 8 类字段你能定位出来，主机适配就已经完成了一半。

## 6. 如何判断一个字段更可能属于哪一层

### 如果它用于路由或识别设备

更可能属于：

- 终端身份层
- 路由上下文层
- KSN / 模式识别层

### 如果它是一段看起来不可读的载荷

更可能属于：

- Data 密文
- PIN Block
- 管理安全块

### 如果它长度固定且出现在报文末尾或安全段中

更可能属于：

- MAC
- 认证结果

### 如果它决定主机应走哪种解析逻辑

更可能属于：

- 模式字段
- profile 字段
- 版本字段

## 7. 和厂商沟通时最该问什么

不要只问：

- 这个模式是什么。

更应该问：

1. 这个模式底层是不是 DUKPT。
2. KSN 在哪里，长度和格式如何。
3. Data / PIN / MAC 是否分开处理。
4. MAC 覆盖哪些字段。
5. 主机应先验 MAC 还是先解密。
6. PIN Block 是否走独立 PIN 域。
7. 是否有不同 profile 或版本兼容差异。
8. 是否有测试向量、样例报文和失败样例。

## 8. 一个推荐的适配流程

你可以把厂商协议适配流程做成下面五步：

1. 先做术语映射。
2. 再做字段分层。
3. 再做安全模式识别。
4. 再做主机侧处理流程图。
5. 最后用测试向量和样例报文验证。

## 9. 一句总结

阅读厂商协议最重要的能力，不是记住更多厂商缩写，而是把任何陌生命名都翻译回“业务字段、密钥上下文、密文对象、完整性对象、主机处理路径”这五个框架里。只要这五层能对齐，厂商协议再陌生，也会变得可控。
