# 厂商协议适配附录：主机解包与处理路径

## 1. 为什么还要单独讲主机侧

很多团队在适配厂商终端时，会把注意力集中在：

- 终端发什么
- 报文字段叫什么

但真实适配难点往往在主机侧，因为主机必须真正知道：

- 先做什么。
- 后做什么。
- 哪些字段先校验。
- 哪些字段进入 HSM。
- 哪些字段绝不能进入普通日志。

所以这一篇专门讲：

- 厂商协议适配到主机时，解包和处理路径应如何分层思考。

## 2. 主机侧最容易犯的错

### 2.1 把整条报文交给一个函数“顺手全处理”

这会导致：

- 路由、MAC 校验、Data 解密、PIN 域处理、日志治理全部混在一起。

### 2.2 先解密再想安全校验

在很多模式下，正确顺序并不一定是“先看能不能解密”，而是先确认：

- 这是哪种模式。
- 需要先验哪些完整性对象。
- 哪些字段能信，哪些字段还不能信。

### 2.3 把 Data 路径和 PIN 路径混成一个通道

这会让 PIN 域和普通数据域的边界模糊化。

## 3. 一个推荐的主机侧分层模型

你可以把主机处理拆成 6 层：

1. 报文接入层
2. 模式识别层
3. 完整性校验层
4. 密钥与路由层
5. Data / PIN 分流处理层
6. 业务结果层

## 4. 第一层：报文接入层

这一层的目标不是理解全部业务，而是先完成：

- 基本报文接收
- 协议外层解析
- 必要的结构校验
- 请求追踪号生成

这一层要特别注意：

- 不要在这里就把全量敏感字段无差别记录到日志。

## 5. 第二层：模式识别层

这一层要回答：

- 这是哪家厂商的哪种模式。
- 是哪种 profile / 版本。
- 是否属于 DUKPT。
- 是否包含 PIN / Data / MAC。

如果这一层识别错了，后面所有处理都可能走错。

## 6. 第三层：完整性校验层

这一层要回答：

- MAC 是否存在。
- MAC 覆盖哪些字段。
- 当前报文是否可信。

这一层的价值在于：

- 尽量在进入更深的敏感处理前，先判断报文有没有被篡改。

## 7. 第四层：密钥与路由层

这一层要回答：

- KSN 是什么。
- 使用哪组 BDK / 用途键 / HSM 域。
- 当前终端属于哪个商户、环境、通道和安全 profile。

这一步通常是主机和 HSM 之间最关键的桥梁。

## 8. 第五层：Data / PIN 分流处理层

这一步非常关键，因为它体现了：

- 不同敏感对象不应进入同一条处理路径。

### Data 路径

通常处理：

- EncryptedDataBlock
- 卡数据相关密文对象

### PIN 路径

通常处理：

- PinBlock
- PIN 域相关对象

### MAC 路径

通常在更前面做校验，但某些系统中也会有独立的认证服务或 HSM 路径承担相关工作。

## 9. 第六层：业务结果层

只有在前面几层完成后，系统才应进入：

- 授权
- 风控
- 状态落库
- 回执生成

这意味着：

- 安全处理不应该被业务层随意绕开。

## 10. 一个推荐的主机处理顺序

你可以把厂商协议主机处理流程记成下面这样：

1. 接收入站报文。
2. 识别厂商模式和 profile。
3. 提取 KSN、模式标识、关键业务字段。
4. 校验 MAC 或其他完整性控制。
5. 选择正确密钥域和 HSM 路径。
6. Data 与 PIN 分流处理。
7. 再进入授权与业务编排。

这套顺序不是唯一实现方式，但它很适合作为架构检查模板。

## 11. 主机实现里建议拆出的几个模块

为了避免逻辑缠绕，建议至少拆出：

- `ProtocolParser`
- `SecurityProfileResolver`
- `MacVerifier`
- `KeyDomainResolver`
- `DataDecryptService`
- `PinDomainService`
- `BusinessOrchestrator`

具体语言和项目结构可以不同，但这种“按安全职责拆模块”的思路非常重要。

## 12. 日志和排障怎么做才不踩坑

主机侧最容易犯的错，是为了排障方便而把以下内容全量打日志：

- 原始报文
- EncryptedDataBlock
- PinBlock
- KSN + 商户映射 + 全量安全 profile

更稳妥的做法是：

- 日志优先记录流程阶段、模式识别结果、终端标识、订单号、错误码、路由域。
- 对高敏安全字段只保留摘要、引用或受控日志。

## 13. 主机适配时最应该做的两张图

### 图一：字段流向图

回答：

- 哪些字段进入哪个模块。

### 图二：安全处理顺序图

回答：

- MAC 在哪里验。
- Data 在哪里解。
- PIN 在哪里处理。
- 业务层何时接手。

只要这两张图画出来，很多原本混乱的适配工作会立刻变得清晰。

## 14. 一句总结

厂商协议适配的真正难点，不是“把报文解析出来”，而是“让主机按正确顺序、正确路径、正确边界去处理不同类型的安全对象”。只要主机侧分层清楚，厂商协议再复杂，也会变成一组可以被拆解、验证和演进的安全处理流程。
