# 03 - 密码学、联机与脚本

## 1. 为什么这一册会让人觉得难

EMV 的密码学部分之所以难，不只是因为有 RSA、会话密钥、证书链，而是因为这些内容和交易控制紧紧交织在一起。

如果前面的交易主线没建立好，就很容易把 ODA、ARQC、ARPC 看成互不相关的几个缩写。

## 2. 先分清三件事

### 2.1 ODA

ODA（Offline Data Authentication）解决的是：

- 这张卡的数据是否可信
- 这张卡是不是被篡改或伪造过

### 2.2 CVM

CVM 解决的是：

- 当前拿卡的人是否通过了应有的持卡人验证

### 2.3 ARQC / ARPC

ARQC / ARPC 解决的是：

- 联机时发卡行如何确认这笔交易来自合法卡片
- 卡片如何确认联机响应来自可信发卡方

## 3. ODA 的三种模式

### 3.1 SDA

- 验证静态数据签名
- 防篡改能力有，但抗复制能力弱

### 3.2 DDA

- 卡片具备动态签名能力
- 可证明卡持有相应私钥，安全性高于 SDA

### 3.3 CDA

- 将动态认证和应用密码结合起来
- 把交易相关数据纳入认证范围，安全性更高

## 4. 为什么要有 CAPK 证书链

终端不可能预装每张卡的公钥，所以需要信任链：

1. 终端预置 CA 公钥（CAPK）
2. CAPK 验证发卡行公钥证书
3. 发卡行公钥再验证 ICC 公钥证书或动态数据
4. 最终完成对卡片静态/动态数据的验证

可以简单理解为：

- CAPK 是信任根
- Issuer Public Key 是中间层
- ICC Public Key 或动态签名数据是卡片层

## 5. ARQC / ARPC 到底在做什么

### 5.1 ARQC

ARQC 是卡片根据交易关键字段生成的授权请求密码。其目的是：

- 让主机确认交易未被篡改
- 让主机确认卡片处于合法状态
- 把 ATC、IAD、风险状态带给发卡行

### 5.2 ARPC

ARPC 是发卡行对 ARQC 的回应，用于让卡片确认：

- 这次联机响应确实来自可信发卡方
- 发卡行对这笔交易的授权态度已被安全地带回卡片

## 6. Field 55 是什么

Field 55 是 ISO8583 中承载 EMV TLV 的关键字段，常用于把芯片交易关键对象打包上送主机。

典型包含：

- `82` AIP
- `95` TVR
- `9F10` IAD
- `9F26` AC
- `9F27` CID
- `9F36` ATC
- `9F37` UN
- `9F34` CVM Results

很多“主机说 ARQC 校验失败”的问题，最后会回到 Field 55 的组包质量上。

## 7. Issuer Script 为什么重要

Issuer Script 是发卡行在联机返回时下发给卡片的远程命令。

它通常用于：

- 更新卡片参数
- 同步风险状态
- 锁定或调整应用行为

常见模板：

- `71`：通常理解为最终 AC 前执行
- `72`：通常理解为最终 AC 后执行

## 8. 这一册最重要的工程认知

### 8.1 ODA 失败不等于只有密钥问题

常见根因包括：

- CAPK 缺失
- RID / Index 不匹配
- Issuer/ICC 证书恢复失败
- 哈希输入顺序错误
- CDA 响应模板解析错误

### 8.2 主机批准不等于卡片最终批准

这是最常见的误解之一。

因为联机批准后：

- 还可能要做 ARPC 验证
- 还可能执行 Issuer Script
- 二次 Generate AC 仍可能返回 AAC

### 8.3 Field 55 不是“随便拼几个 Tag”

它必须满足：

- 标签正确
- 编码正确
- 长度正确
- 来源正确
- 顺序符合主机约定

## 9. 学这一册时的建议方法

### 先看关系，不先看算法细节

先理解：

- ODA 解决卡可信
- ARQC/ARPC 解决联机可信
- Script 解决主机对卡的后续控制

### 再看对象链路

- CAPK → Issuer Key → ICC Key
- ARQC → Host Verify → ARPC → Card Verify
- Field 55 → Host → Script → 二次 AC

## 10. 这一册学完后，你应该达到什么程度

读完后，你应该能：

- 区分 ODA、CVM、ARQC/ARPC 各自解决的问题
- 用自己的话解释 CAPK 证书链
- 知道 Field 55 为什么经常是联机问题根因
- 理解为什么联机批准后卡片仍可能最终拒绝
