# 04 - 内核架构与工程实践

## 1. 为什么规范理解完还不够

很多人读完 EMV 规范后，还是写不好 Kernel。原因是：

- 规范描述的是行为要求
- 工程实现需要考虑状态机、模块边界、日志、持久化、异常恢复

所以从“会解释规范”到“能实现内核”，中间还差一层软件架构设计。

## 2. POS Kernel 到底是什么

POS Kernel 可以理解为：

- 管理交易状态机的核心引擎
- 调度 APDU、TLV、风险控制、CVM、AC、Script 的协议核心
- 连接终端设备能力与上层业务的中间层

它不应该承担：

- 收银 UI 细节
- 清算与批结算
- 复杂商户业务规则

## 3. 推荐的软件分层

### 3.1 驱动与设备层

负责：

- 读卡器
- 非接控制器
- PIN Pad
- 安全硬件

### 3.2 APDU 与协议层

负责：

- APDU 收发
- T=0/T=1、ISO 14443
- 超时、重试、链路稳定性

### 3.3 Kernel 核心层

负责：

- 交易状态机
- TLV 仓库
- 应用选择
- CVM
- 风险管理
- Generate AC
- Issuer Script

### 3.4 业务编排层

负责：

- 金额输入
- 交易类型
- 联机通信
- 冲正
- 签购单
- 业务 UI

## 4. 推荐的核心模块

| 模块 | 职责 |
| --- | --- |
| Kernel Facade | 对上层暴露统一接口 |
| Transaction State Machine | 管理交易阶段与迁移 |
| APDU Service | 封装 SELECT、GPO、READ RECORD、GENERATE AC 等 |
| TLV Repository | 统一存放 card、terminal、computed、host 标签 |
| Rule Engines | 封装 Application Selection、CVM、Risk、Action Analysis |
| Crypto Service | ODA、随机数、安全硬件调用、密钥相关数据处理 |
| Terminal Adapter | 适配 Reader、PIN Pad、UI、Host、存储 |

## 5. 为什么状态机驱动很重要

EMV 交易天然就是强顺序、强上下文的流程。

如果没有显式状态机，常见问题会很快出现：

- 联机返回后找不到正确上下文
- cancel / timeout / card remove 后状态混乱
- 脚本与二次 AC 时序错误
- 非接 Outcome 与 UI 回调脱节

所以内核最好以状态机作为中心骨架，而不是堆很多布尔变量。

## 6. 为什么 TLV 仓库要中心化

EMV 交易过程中，同一个对象可能来自：

- 卡片
- 终端参数
- 终端计算
- 主机回包

如果这些对象散落在局部变量里，后果通常是：

- 很难复盘
- 很难组 Field 55
- 很难看清对象来源
- 很难做日志快照

所以一个统一的 TLV Repository 对工程质量影响很大。

## 7. 品牌差异为什么要插件化

公共 EMV 骨架和品牌私有差异不应该混在一起。

建议做法：

- 公共层：交易主线、TLV 仓库、通用 ODA/CVM/Risk 框架
- 品牌层：AID 路由、私有标签解释、专有 CVM / 风险规则、Field 55 细节

这样更容易：

- 加新品牌
- 做回归测试
- 降低不同品牌间互相污染

## 8. 工程实现中的高优先级问题

### 8.1 日志

日志至少要能回答：

- 当前处于哪个状态
- 刚发了什么 APDU
- 收到了什么响应
- 哪些关键标签被更新了
- 哪些 TVR bit 被置位了
- 为什么决定联机 / 拒绝 / 继续

### 8.2 持久化

需要重点考虑：

- 掉电恢复
- 联机前后断点
- 冲正所需数据
- 日志可追踪性

### 8.3 参数治理

重点参数包括：

- AID
- CAPK
- TAC
- Floor Limit
- CVM 相关限额
- Reader limit / contactless capability

这些参数应支持版本化、远程下载和回滚。

## 9. 初学者最容易写错的地方

1. 把品牌差异硬编码到通用流程里
2. 把终端 UI 和内核决策耦合在一起
3. 没有中心化的 TLV 管理
4. 没有结构化日志
5. 只关注交易跑通，不考虑异常恢复和可维护性

## 10. 这一册学完后，你应该达到什么程度

读完后，你应该能：

- 说清一个 POS Kernel 的推荐分层
- 理解为什么状态机、TLV 仓库和插件化对 EMV 特别重要
- 把规范术语映射到真实模块职责和接口设计上
