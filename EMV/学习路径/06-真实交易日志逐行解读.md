# 06 - 真实交易日志逐行解读

## 1. 为什么要专门学“读日志”

很多人把 EMV 学到一定阶段后，会卡在一个问题上：

- 规范能看懂
- 流程也知道
- 但一拿到现场日志，还是不知道从哪里下手

这是因为日志阅读不是“再背一次规范”，而是要学会把下面四件事连起来：

1. 当前交易走到了哪一步
2. 当前这一步产出了哪些关键对象
3. 当前这一步是谁在做决策
4. 当前异常是输入错误、执行失败，还是策略命中

## 2. 先给自己一个固定阅读顺序

无论日志多长，都建议按这个顺序看：

1. 接口类型：接触还是非接
2. 激活链路：ATR / ATS 是否正常
3. 应用选择：最终 AID、Kernel ID、Label
4. GPO：PDOL 输入、AIP、AFL
5. 读记录：关键对象是否读全
6. ODA / CVM / Risk：TVR、TSI、9F34 如何变化
7. 第一次 Generate AC：CID 是什么
8. 联机：Field 55、ARC、ARPC、Script
9. 二次 Generate AC：最终 CID 和卡片态度是什么
10. 终端 Outcome：UI 显示是否与卡片结果一致

## 4. 逐段阅读示范

### 4.1 交易启动与接口识别

```text
+[TXN] start type=SALE amount=000000010000 currency=0156
+[IF] contact card inserted
+[ICC] ATR=3B 67 00 00 ...
+```

这三行主要回答：

- 这是一笔什么交易
- 金额和币种是否正确
- 当前是接触式，不是非接
- 卡片链路是否已建立

如果这里 ATR 异常，后面很多分析都没有意义。

### 4.2 应用选择

```text
+[APDU] SELECT PSE -> 9000
+[APDU] SELECT AID=A0000000031010 -> 9000
+[SEL] finalAid=A0000000031010 label=VISA CREDIT priority=01
+```

这一段主要回答：

- 目录选择是否正常
- 最终是否选中了预期 AID
- 后续应该由哪个品牌逻辑接管

如果最终 AID 错了，后面所有 TVR、Field 55、联机差异都可能只是“选错应用”的连锁反应。

### 4.3 GPO：能力与读卡计划

```text
+[DOL] PDOL input=...
+[APDU] GPO -> 770E82025800940810010100
+[GPO] AIP=5800 AFL=10010100
+```

这一段要回答：

- PDOL 是否已按要求填充
- 卡片支持什么能力
- 终端接下来要读哪些记录

这里的重点不是把 `5800` 每个 bit 全背下来，而是先问：

- 卡支持 ODA 吗
- 卡支持 CVM 吗
- 卡是否期望终端做风险管理
- AFL 是否合理

### 4.4 READ RECORD：关键对象入仓

```text
+[APDU] READ RECORD SFI=2 REC=1 -> 9000
+[TLV] 5A=... 5F24=... 8C=... 8D=... 8E=...
+```

这一步是在确认：

- 卡片关键数据是否已经读到
- PAN、有效期、CDOL1、CDOL2、CVM List 等对象是否可用

如果这里 `8C`、`8D`、`8E` 等对象缺失，后面 Generate AC、CVM、联机都可能出现异常。

### 4.5 ODA / TSI / TVR：本地验证阶段

```text
+[ODA] DDA supported, authentication passed
+[TSI] 6800
+[TVR] 0000000000
+```

这一段要联合看：

- ODA 是否执行了
- TSI 是否表明该步骤确实执行过
- TVR 是否显示失败或异常

如果日志写了 “ODA passed”，但 TSI 没体现对应步骤，说明日志和状态记录可能不一致。

### 4.6 CVM 与风险阶段

```text
+[CVM] result=NO_CVM 9F34=1E0300
+[RISK] floorLimitExceeded=false randomSelected=false
+[TVR] 0000000000
+```

这一段要回答：

- 最终执行了哪种 CVM
- `9F34` 是否和 UI 现象一致
- 本地风险是否触发联机倾向
- TVR 是否因此增加新 bit

如果 UI 显示输入过 PIN，但 `9F34` 是 No CVM，要优先相信内核结果并反查 UI 或设备交互。

### 4.7 TAA 与第一次 Generate AC

```text
+[TAA] decision=GO_ONLINE
+[APDU] GENERATE AC(1) -> ...
+[AC1] CID=ARQC ATC=0012 AC=1122334455667788
+```

这一步是卡片真正开始表态：

- TAA 为什么决定联机
- 第一次 AC 返回的是 AAC、TC 还是 ARQC
- ATC 是否合理递增

如果 TAA 说可以离线，但 AC1 返回 ARQC，也不一定矛盾，因为卡片仍有自己的决策权。

### 4.8 联机与脚本

```text
+[ISO8583] F55=...
+[HOST] response=00 arpc=1234567890ABCDEF script71=... script72=...
+[SCRIPT] tag71 exec result=9000
+```

这一段关注：

- Field 55 是否上送了关键标签
- 主机是否批准
- ARPC 是否回传
- Script 是否执行成功

这里最容易出现的误解是：主机响应 `00` 就认为交易已经成功。实际上还没有，卡片还没做最终确认。

### 4.9 二次 Generate AC 与最终结果

```text
+[APDU] GENERATE AC(2) -> ...
+[AC2] CID=TC ATC=0012 AC=AABBCCDDEEFF0011
+[OUTCOME] APPROVED
+```

这一段才是最终闭环：

- 二次 AC 返回 TC，说明卡片接受本次交易完成
- 如果这里返回 AAC，那么即使主机批准，终端也不能简单视为成功

## 5. 一个常见异常案例：主机批准但交易失败

### 5.1 示例日志

```text
+[HOST] response=00 arpc=1234567890ABCDEF script72=...
+[SCRIPT] tag72 exec result=6985
+[TVR] 0000000004
+[APDU] GENERATE AC(2) -> 770A9F2701009F360200129F2608FFEEDDCCBBAA9988
+[AC2] CID=AAC
+[OUTCOME] DECLINED
+```

### 5.2 应该如何读

这段日志说明：

- 主机批准了
- 但 Script 72 执行失败
- 二次 Generate AC 返回 AAC
- 最终交易应视为失败或卡片拒绝

### 5.3 初步结论

这里不能说“主机没问题，所以终端错了”，更准确的说法应是：

- 主机授权批准
- 但联机后处理没有完成卡片闭环
- 问题可能在脚本内容、脚本时机、ARPC 或二次 AC 输入

## 6. 一个常见异常案例：GPO 失败

### 6.1 示例日志

```text
+[DOL] PDOL input=9F02060000000100009F1A020156950500000000005F2A0201569A03260316...
+[APDU] GPO -> 6A80
+[OUTCOME] TERMINATED
+```

### 6.2 应该先怀疑什么

优先怀疑：

- DOL 长度不匹配
- 某个字段编码方式错误
- PDOL 缺少卡要求的数据对象
- 非接场景缺少 TTQ 等必要字段

这类问题通常不需要先去怀疑主机或证书链。

## 7. 一个最实用的日志阅读框架

以后只要拿到日志，就沿着下面 3 条线索走：

### 第一条：阶段线

交易目前走到了哪一步？

### 第二条：对象线

这一阶段产出了哪些关键对象？

### 第三条：决策线

是谁根据哪些对象做了什么决定？

只要这三条线不乱，EMV 日志就会从“很玄学”变成“可定位”。

## 8. 初学者练习建议

建议你自己拿一笔真实日志，按下面格式做训练：

1. 标出 12 步交易中的位置
2. 每一步写出关键对象
3. 标出 TVR / TSI / CID / 9F34 的变化
4. 标出谁做了联机 / 拒绝 / 通过决策
5. 最后总结问题根因最可能在哪一段

## 9. 这一册学完后，你应该达到什么程度

读完后，你应该能：

- 看懂一笔接触式交易的基本日志骨架
- 知道主机批准为什么不一定等于最终成功
- 知道 GPO、ODA、Field 55、Script、二次 AC 各自在哪一段看
