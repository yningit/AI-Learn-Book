# 07 - 品牌差异入门

## 1. 为什么一定要讲品牌差异

很多人刚接触 EMV 时，会先学到一个“通用骨架”，然后以为所有品牌都差不多。

这在入门阶段是对的，但如果真正开始开发或联调，你很快就会发现：

- 不同品牌的非接入口不一样
- 不同品牌对 Field 55、IAD、CVR 的要求不同
- 不同品牌对 CVM、限额、Outcome、脚本、联机后处理存在差异

所以一个成熟的工程认知应该是：

- **先用通用 EMV 骨架理解交易主线**
- **再把品牌差异当成插件化扩展来理解**

## 2. 什么是“通用骨架”，什么是“品牌差异”

### 2.1 通用骨架

这些内容通常可看作 EMV 共性：

- 应用选择
- GPO / AIP / AFL
- READ RECORD
- ODA
- CVM
- Risk Management
- Action Analysis
- Generate AC
- ARQC / ARPC
- Issuer Script

### 2.2 品牌差异

这些内容更容易出现 scheme-specific 差异：

- 候选应用路由方式
- 非接 Entry Point 与 Kernel Identifier
- TTQ / Reader limit / CVM limit 规则
- IAD / CVR 结构
- Field 55 标签要求与排序
- Outcome 模型与 UI 建议
- 脚本处理细节

## 3. 为什么工程上不能把品牌差异写进通用层

因为这样很容易导致：

- 一个品牌的修改影响另一个品牌
- 代码中充满条件分支
- 回归测试难以覆盖
- 排错时很难知道是通用逻辑问题还是品牌逻辑问题

更好的做法是：

- 通用层：保留 EMV 主流程骨架
- 品牌层：通过 profile / plugin 承载品牌差异

## 4. 如何理解 Visa、Mastercard、UnionPay 的差异

这一节不追求规范级细节，而是建立一个“工程视角”的对比框架。

### 4.1 非接入口与路由

| 维度 | 常见差异点 |
| --- | --- |
| 候选应用选择 | 候选 AID 的处理、优先级和用户选择方式可能不同 |
| Kernel 路由 | 非接通常依赖 Kernel ID / 方案定义路由规则 |
| Outcome | 不同品牌对最终 UI / Reader 提示的抽象方式不同 |

### 4.2 风险与限额

| 维度 | 常见差异点 |
| --- | --- |
| Reader 限额 | 各品牌对 reader contactless limit、floor limit、CVM limit 的用法和配置可能不同 |
| Online 倾向 | 某些品牌更依赖终端能力位和 Reader 配置决定是否走联机 |
| CVM 决策 | 小额免密、签名、联机 PIN 的规则细节可能不同 |

### 4.3 联机字段与发卡行数据

| 维度 | 常见差异点 |
| --- | --- |
| IAD / CVR | 结构通常是品牌自定义，不能用统一硬编码解释 |
| Field 55 | 必传标签、排序、格式要求可能不同 |
| ARQC/ARPC 数据解释 | 核心思路一致，但细节字段和算法上下文可能不同 |

## 5. 一个正确的工程心智模型

不要问：

- “到底谁的规则才是对的？”

更好的问法是：

- “这一段是 EMV 通用规则，还是品牌扩展规则？”

这会极大提升你读规范和读代码的效率。

## 6. 适合工程实现的抽象方式

建议在代码设计上做如下划分：

### 6.1 通用层负责

- 交易状态机
- TLV 仓库
- APDU 编排框架
- 通用风险与动作分析骨架
- 通用日志和调试快照

### 6.2 品牌层负责

- AID / Kernel ID / Route profile
- 私有标签解释
- 私有 CVM / 风险 / Outcome 规则
- Field 55 细节映射
- IAD / CVR 解释

## 7. 初学者最容易踩的坑

### 7.1 把通用 EMV 教材里的说法当成所有品牌都完全一致

这是最常见误区。很多教材为了帮助入门，会先讲通用骨架，但真实项目里品牌差异很快就会出现。

### 7.2 试图用一套硬编码逻辑解释所有 IAD

这通常会导致：

- 主机联调困难
- ARQC 校验问题难定位
- 品牌切换时大量改动

### 7.3 把非接路由写死

非接场景下，Entry Point、候选应用和品牌规则都可能影响最终路由，写死会让后期维护非常痛苦。

## 8. 一个实用的品牌差异阅读框架

以后遇到某个品牌的专有规则时，可以固定问下面 5 个问题：

1. 这是通用 EMV 规则，还是品牌扩展？
2. 它影响交易的哪个阶段？
3. 它影响哪些标签或对象？
4. 它应该落在通用层还是品牌插件层？
5. 它会影响哪些日志、测试和联机字段？

## 9. 这一册学完后，你应该达到什么程度

读完后，你应该能：

- 理解“先学通用骨架，再学品牌差异”这条正确路线
- 知道为什么品牌逻辑必须做工程隔离
- 理解品牌差异主要会落在哪些对象和模块上
