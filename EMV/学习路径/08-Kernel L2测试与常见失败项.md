# 08 - Kernel L2 测试与常见失败项

## 1. 什么是 Kernel L2 测试

如果把 EMV 终端分层理解，大致可以这样看：

- L1：更多偏底层物理链路、读卡器和协议传输能力
- L2：更多偏交易流程、内核行为、EMV 规则执行和品牌要求实现
- L3：更多偏终端业务集成、主机联调、真实业务场景

所以，Kernel L2 测试的核心目标不是“机器能读卡”，而是：

- 内核是否按规范执行交易流程
- 关键对象、位图、APDU、结果是否符合要求
- 品牌要求和异常路径是否被正确实现

## 2. 为什么 L2 测试对 EMV Kernel 很关键

很多问题在开发环境看起来“交易能跑通”，但在 L2 测试里会暴露出来，因为测试关心的不只是结果，还包括：

- 是否按规定步骤执行
- 是否设置了正确的 TVR / TSI / CVM Results
- 是否在正确时机发送正确 APDU
- 是否正确处理例外和异常路径
- 是否对品牌要求做了正确分支

也就是说，L2 测试是在验证：**你的实现是否真的符合内核规范，而不是仅仅看起来能交易。**

## 3. L2 测试通常会覆盖什么

### 3.1 交易主流程

- 正常消费
- 脱机批准
- 脱机拒绝
- 联机批准
- 联机拒绝
- 联机不可达 / 超时

### 3.2 ODA 相关

- SDA 成功 / 失败
- DDA 成功 / 失败
- CDA 成功 / 失败
- ODA 未执行 / 数据缺失

### 3.3 CVM 相关

- Offline PIN
- Online PIN
- Signature
- No CVM
- CVM 失败
- PIN bypass / PIN pad 异常

### 3.4 风险与动作分析相关

- Floor Limit 命中
- Random Selection 命中
- TAC / IAC 不同组合
- Processing Restrictions 异常

### 3.5 联机与后处理相关

- ARQC / ARPC 路径
- Issuer Script 成功 / 失败
- 二次 Generate AC 结果差异
- 主机批准但卡片最终 AAC

## 4. 测试时最常见的失败类型

### 4.1 流程顺序错误

表现：

- 某一步本该执行却没有执行
- APDU 发送顺序错误
- 脚本时机错误
- 二次 Generate AC 时机错误

### 4.2 对象或位图错误

表现：

- TVR 未置位或置错位
- TSI 没反映真实执行步骤
- `9F34` 不符合实际 CVM 流程
- CID 解释错误

### 4.3 DOL / TLV 组包错误

表现：

- GPO 返回 `6A80` 或 `6985`
- Generate AC 输入错误
- Field 55 编码错误

### 4.4 品牌差异处理错误

表现：

- 某品牌通过，另一个品牌失败
- 非接路由错误
- Outcome 不符合品牌要求
- IAD / CVR / 专有标签解释错误

## 5. 一套实用的 L2 测试准备清单

### 5.1 代码层准备

- 关键 APDU 日志可开关
- 关键 TLV 快照可导出
- TVR / TSI / 9F34 / CID 可结构化打印
- 状态迁移可追踪
- 联机映射可回放

### 5.2 参数层准备

- AID 参数清单
- CAPK 清单
- TAC / Floor Limit / Reader limit 清单
- 非接参数与 CVM limit 清单
- 测试环境与生产环境参数隔离

### 5.3 测试协作准备

- 测试 case 列表与预期结果
- 日志采集方式统一
- 失败 case 的问题单模板
- 主机联调联系人和字段说明

## 6. 遇到 L2 失败时该怎么定位

建议按下面顺序定位：

1. 先定位交易阶段：失败发生在应用选择、GPO、ODA、CVM、AC1、联机、脚本还是 AC2
2. 再定位对象：AIP、AFL、TVR、TSI、9F34、CID、Field 55、ARPC、Script
3. 再定位责任边界：读卡器 / 内核 / 主机 / 参数 / 品牌配置
4. 最后再回到规范条款和 case 预期做比对

这样比直接逐行看长日志更高效。

## 7. 一个高频失败案例清单

| 失败现象 | 常见根因 |
| --- | --- |
| GPO 失败 | PDOL 组装错误、长度错误、缺少 TTQ/金额/日期等字段 |
| ODA 失败 | CAPK 缺失、证书链恢复错误、认证数据顺序错误 |
| CVM case 不通过 | 终端能力配置错误、PIN 设备能力不匹配、CVM List 解释错误 |
| 联机 case 不通过 | Field 55 缺标签、编码错误、ARPC 消费错误 |
| Script case 不通过 | 71/72 时机错误、脚本 APDU 执行失败未正确处理 |
| 非接 case 不通过 | Entry Point 路由错误、Reader 参数不正确、Outcome 不符合品牌要求 |

## 8. 为什么要把 L2 失败沉淀成知识库

L2 失败往往不是“一次性的 bug”，而是：

- 某类对象处理模型有缺陷
- 某类品牌隔离做得不够
- 某类日志不够支撑定位
- 某类参数治理有漏洞

所以建议每次失败都沉淀：

- case 名称
- 失败现象
- 根因分类
- 修复点
- 回归范围
- 是否需要新增自动化测试

## 9. 初学者最容易误解的地方

### 9.1 “测试卡能交易就说明内核没问题”

不对。L2 更关注行为是否符合规范，而不只是是否能成交。

### 9.2 “L2 失败一定是代码 bug”

也不对。很多失败来自：

- 参数错
- CAPK 版本错
- 主机测试配置错
- 品牌测试环境假设不一致

### 9.3 “只要修复当前 case 就行”

也不对。很多 case 共享同一条交易主线，修一个点很可能影响别的路径。

## 10. 对团队协作的建议

- 开发、测试、主机联调、参数管理最好共享同一套问题分类模型
- 所有失败 case 都应能定位到：阶段、对象、责任边界
- 修复后最好补自动化回归或最少补问题库记录

## 11. 这一册学完后，你应该达到什么程度

读完后，你应该能：

- 理解 L2 测试到底在测什么
- 知道 L2 失败最常落在哪几类问题上
- 能根据失败现象快速缩小定位范围
- 知道为什么日志、参数和品牌隔离对 L2 特别关键
