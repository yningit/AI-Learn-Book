# 09 - 参数与配置治理

## 1. 为什么 EMV 项目一定要重视参数治理

很多 EMV 问题表面看像代码 bug，实际上根因往往是参数或配置治理混乱，例如：

- AID 配错导致应用选错
- CAPK 版本旧导致 ODA 失败
- TAC / Floor Limit 配置不一致导致交易路径异常
- 非接 Reader 参数错误导致频繁插卡或错误 CVM 决策

所以在 EMV 项目里，参数治理不是“运维问题”，而是内核工程质量的一部分。

## 2. 典型要治理的参数有哪些

### 2.1 AID 参数

通常包括：

- AID
- Application Label
- 优先级
- 是否允许部分匹配
- 接触 / 非接支持标识
- 所属品牌或 Kernel profile

### 2.2 CAPK 参数

通常包括：

- RID
- Index
- Key modulus / exponent
- 有效期
- 版本或来源标识

### 2.3 风险参数

通常包括：

- Floor Limit
- Random Selection 相关阈值
- TAC-Denial / TAC-Online / TAC-Default
- Exception file 相关策略

### 2.4 非接参数

通常包括：

- Reader limit
- CVM limit
- Contactless capability / TTQ 相关配置
- Entry Point / 路由 profile

## 3. 为什么参数不能散落在代码和多张表里

如果参数没有统一治理，常见后果是：

- 不同终端配置不一致
- 同一品牌在不同版本间行为漂移
- 出问题时不知道当前生效的是哪版参数
- 测试环境与生产环境串用
- 开发只能靠猜当前配置

所以参数治理的第一原则是：**统一来源、统一版本、统一可追踪。**

## 4. 一个推荐的参数分层模型

### 4.1 全局层

适用于所有终端和商户的基础参数，例如：

- 公共 CAPK
- 公共品牌 profile
- 默认终端能力模板

### 4.2 品牌层

按品牌或 kernel profile 维护的参数，例如：

- AID 集合
- 品牌风险参数
- Field 55 相关映射配置
- 非接 Reader 配置

### 4.3 终端 / 商户层

按终端型号、商户、国家、业务场景差异化维护的参数，例如：

- Floor Limit
- 联机策略
- 设备能力开关
- 非接限额

## 5. 一个推荐的版本管理模型

EMV 参数至少应做到：

- 每套参数有唯一版本号
- 支持发布时间、变更说明、审批记录
- 支持灰度发布
- 支持回滚
- 支持测试环境 / 预发环境 / 生产环境隔离

## 6. AID 和 CAPK 为什么最需要严格管理

### 6.1 AID 管理不好会发生什么

- 应用选择错误
- 候选应用优先级异常
- 非接路由错误
- 品牌逻辑进入错误分支

### 6.2 CAPK 管理不好会发生什么

- ODA 失败
- 某些卡在某些终端正常、在另一些终端失败
- 测试环境通过、生产环境失败
- 测试和生产参数不一致导致难以复现

## 7. TAC / Floor Limit / Reader Limit 为什么要联动看

这些参数会直接影响：

- 交易是否倾向联机
- 交易是否可能被拒绝
- 非接是否允许继续
- 小额免密和 CVM 决策是否合理

所以它们不能单独看，而要作为一组“交易路径控制参数”一起管理。

## 8. TMS 下载为什么是工程重点

参数治理最终通常要落到 TMS（Terminal Management System）或等价配置平台。这里最重要的不是“能下载”，而是：

- 下载内容可验证
- 下载结果可确认
- 生效版本可追踪
- 失败可回滚
- 多终端批量下发可控

## 9. 一个推荐的参数治理流程

1. 参数设计与评审
2. 测试环境下发与验证
3. 联调验证
4. 小规模灰度
5. 批量发布
6. 发布后监控
7. 异常时快速回滚

## 10. 常见参数治理问题

| 问题 | 常见后果 |
| --- | --- |
| AID 集合不一致 | 不同终端选择不同应用 |
| CAPK 版本旧 | ODA 失败 |
| TAC / Floor Limit 配置漂移 | 同类交易路径不一致 |
| Reader/CVM limit 不合理 | 非接频繁插卡或 CVM 结果异常 |
| 测试/生产环境混用 | 问题难复现、联调混乱 |
| 参数无版本追踪 | 无法确认问题发生时实际生效配置 |

## 11. 参数治理和日志为什么要联动

一个成熟的 EMV 系统，日志最好能打印：

- 当前命中的参数版本
- 当前命中的 AID profile / 品牌 profile
- 当前命中的 CAPK 标识
- 关键 Reader / Floor / TAC 配置摘要

这样排查问题时，才能迅速分辨：

- 是代码逻辑有问题
- 还是配置本身就不对

## 12. 对团队的建议

### 开发侧

- 不要把参数写死到代码里
- 为关键参数提供结构化快照输出

### 测试侧

- case 执行前确认参数版本
- case 失败时记录实际生效参数

### 运维 / 配置侧

- 建立参数审批、发布、回滚和环境隔离机制
- 对高风险参数变更建立双人复核

## 13. 这一册学完后，你应该达到什么程度

读完后，你应该能：

- 理解为什么参数治理是 EMV 工程的一部分
- 知道 AID、CAPK、TAC、Floor Limit、Reader limit 为什么关键
- 知道 TMS 下载不能只关注“能不能下发”，还要关注版本、追踪和回滚
