# 10 - 品牌差异对照版

## 1. 这份文档的目的

前面的 [07-品牌差异入门](07-品牌差异入门.md) 解决的是“为什么会有品牌差异、工程上该怎么隔离”。

这份文档进一步回答：

- 不同品牌的差异大致会落在哪些地方
- 这些差异对终端内核实现意味着什么
- 开发、测试、联调时应该优先关注什么

> 说明：本篇聚焦工程视角的“差异类型”和“实现落点”，不是品牌规范逐条翻译。具体行为仍需以对应 scheme 规范和认证要求为准。

## 2. 一个正确的前提

先记住两句话：

1. **EMV 通用骨架是共性的基础。**
2. **品牌差异主要体现在入口、对象解释、联机要求、Outcome 和测试认证细节上。**

如果没有先学会通用骨架，就很容易把品牌差异看成“每家都完全不同”。

## 3. 对照时最有用的 5 个维度

建议以后比较任何品牌时，都固定按这 5 个维度看：

1. 入口与路由
2. 关键对象与数据结构
3. CVM / 风险 / Outcome 规则
4. 联机字段与主机协作
5. 测试与认证关注点

## 4. 品牌差异对照总表（工程视角）

| 维度 | Visa | Mastercard | UnionPay | 工程实现建议 |
| --- | --- | --- | --- | --- |
| 入口与路由 | 常强调接触/非接应用路径与品牌 profile 清晰隔离 | 非接 kernel 行为和 outcome 模型要求通常较明确 | 本地化场景、受理环境和参数组合更常见 | 用 Entry Point + scheme plugin 模型统一路由 |
| IAD / CVR | 品牌私有解释较明显 | 品牌私有解释较明显 | 品牌私有解释较明显 | 不要在通用层硬编码解释 IAD/CVR |
| CVM / 限额 | 非接小额、CVM 规则受 reader 参数和品牌规则共同影响 | Contactless 体验和 outcome 抽象通常较强 | 本地参数策略和受理场景差异更常见 | 把 CVM 和 limit 决策做成 profile 驱动 |
| 联机字段 | Field 55 标签集和主机约定较关键 | 联机后处理和 outcome 一致性要求高 | 主机与终端参数一致性很关键 | 建立 scheme-specific mapping 层 |
| 认证与测试 | 规范细节和认证案例较细 | 测试 case 和 outcome 细节要求常较多 | 本地测试环境与品牌规则结合更紧 | 测试 case、日志、参数都要按品牌隔离 |

## 5. 入口与路由差异

### 5.1 通用理解

所有品牌都离不开：

- 候选应用发现
- AID 选择
- 接触 / 非接入口识别
- Kernel 接管

### 5.2 品牌差异通常体现在哪

- 非接是否依赖更明确的 Kernel ID / Entry Point 路由
- 多应用冲突时的优先级和选择行为
- Outcome 对 UI / Reader 的抽象方式

### 5.3 工程建议

- 路由逻辑不要散落在品牌实现里
- 把候选应用发现、路由决策和品牌执行分成三层
- 统一输出 `KernelRouteDecision` 和 `KernelOutcome`

## 6. 关键对象差异

### 6.1 通用对象

这些对象基本所有品牌都会涉及：

- AID
- AIP
- AFL
- TVR
- TSI
- CID
- ATC
- `9F34`

### 6.2 品牌私有解释最常落在哪

- IAD
- CVR
- 某些品牌专有 Tag
- Outcome 结构
- Reader / TTQ / CVM limit 的解释方式

### 6.3 工程建议

- 通用对象统一建模
- 私有对象留给品牌 profile 解释
- 不要把某一家品牌的 IAD 结构当成全局模板

## 7. CVM、风险和 Outcome 的差异

### 7.1 通用骨架仍然一致

所有品牌最终都要回答：

- 是否需要做 CVM
- 是否应联机
- 是否应拒绝
- 最终交易如何对用户呈现

### 7.2 差异通常落在哪

- No CVM / Signature / PIN 的优先策略细节
- contactless limit / CVM limit / reader limit 的实际用法
- Outcome 是否以更强的 reader 指令模型暴露给上层

### 7.3 工程建议

- 把 CVM / Risk / Outcome 的品牌差异放在策略层
- 公共状态机只管理交易阶段，不管理品牌细节

## 8. 联机与主机协作差异

### 8.1 通用部分

通用部分仍是：

- Field 55 组包
- ARQC 上传
- 主机验证
- ARPC / Script 回传
- 二次 AC 闭环

### 8.2 差异常见在

- Field 55 必传标签集合
- IAD / CVR 的解释与主机风控使用方式
- 授权后脚本处理差异
- 某些标签是否要求上送、如何编码

### 8.3 工程建议

- 建立 `EMV Context -> Host Mapping` 的品牌层映射
- 保留主机联调快照，按品牌展示关键标签
- 不同品牌的 host mapping 不要共用一套硬编码 switch

## 9. 测试与认证关注点差异

### 9.1 为什么品牌差异会先在测试里暴露

因为品牌差异最容易体现在：

- 特定 case 的预期 TVR / TSI / Outcome
- 特定对象的上送与解释
- 特定失败路径是否按规范处理

### 9.2 工程建议

- 按品牌维护测试 case 集合
- 按品牌维护日志检查项
- 按品牌维护参数 profile
- L2 / 品牌认证问题要能追踪到具体 scheme profile

## 10. 一个建议的实现结构

```text
Entry Point / App Selection
  -> Common EMV Flow Engine
      -> Scheme Profile / Plugin
          -> Scheme Object Parser
          -> Scheme Risk / CVM / Outcome Rules
          -> Scheme Host Mapping
          -> Scheme Test Profile
```

## 11. 初学者最容易误解的地方

### 11.1 “品牌差异”意味着所有内容都不同

不对。通用骨架仍然很重要，也是最先要学会的部分。

### 11.2 只要交易过了，品牌实现就没问题

不对。很多品牌差异会在：

- TVR / TSI 细节
- Outcome 呈现
- Field 55 标签要求
- 脚本处理
- 认证 case

中暴露出来。

### 11.3 用大量 `if brand == ...` 就能解决问题

短期看似能解决，长期通常会导致公共层失控。

## 12. 对开发团队的建议

- 先统一共性骨架，再落品牌插件
- 按品牌隔离参数、日志、host mapping、测试 case
- 任何品牌差异需求都先问：它应该落在通用层，还是 profile 层？

## 13. 这一册学完后，你应该达到什么程度

读完后，你应该能：

- 知道比较品牌差异时应看哪些维度
- 知道为什么品牌差异要做工程隔离
- 知道品牌问题通常会落在哪些模块和对象上
