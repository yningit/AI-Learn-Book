# 11 - 真实案例库

## 1. 这份案例库怎么用

这份文档不追求“列举所有错误”，而是挑最能帮助你建立排错能力的高频案例。

每个案例都按同一结构组织：

1. 现象
2. 关键日志 / 关键对象
3. 正确分析路径
4. 高概率根因
5. 工程修复建议

建议你把它和 [05-调试、排错与测试](05-调试、排错与测试.md) 以及 [06-真实交易日志逐行解读](06-真实交易日志逐行解读.md) 结合着看。

## 案例 1：`SELECT` 成功，但 `GPO` 失败

### 1. 现象

- 目录选择和 AID 选择都成功
- `GET PROCESSING OPTIONS` 返回 `6A80` 或 `6985`
- 交易在很前面就终止

### 2. 关键日志

```text
+[APDU] SELECT AID=... -> 9000
+[DOL] PDOL input=...
+[APDU] GPO -> 6A80
+```

### 3. 正确分析路径

先不要怀疑主机，也不要先怀疑证书链。优先看：

1. PDOL 是否完整
2. DOL 中每个字段长度是否正确
3. 编码是否正确（BCD / Binary / Numeric）
4. 非接场景是否遗漏 TTQ 等必要字段

### 4. 高概率根因

- PDOL 组装顺序错误
- 某字段长度不足或过长
- 把十六进制值当 ASCII 发给卡
- 缺少卡要求的对象

### 5. 工程修复建议

- 建立统一 DOL Builder
- 每个字段显式声明长度、编码、默认值来源
- 对 GPO 输入做调试快照打印

## 案例 2：ODA 失败，但卡明明支持 DDA / CDA

### 1. 现象

- AIP 显示支持 DDA 或 CDA
- READ RECORD 也返回了认证相关对象
- 但 TVR 显示 ODA failed，或日志显示动态认证失败

### 2. 关键日志

```text
+[GPO] AIP=...
+[APDU] READ RECORD -> 9000
+[ODA] DDA verify failed
+[TVR] ...
+```

### 3. 正确分析路径

按顺序排：

1. AIP 是否真的声明支持对应模式
2. AFL 是否把认证相关记录读全
3. RID + Index 是否命中本地 CAPK
4. Issuer / ICC 证书恢复是否成功
5. 哈希输入和拼接顺序是否正确
6. CDA 场景下响应模板解析是否正确

### 4. 高概率根因

- CAPK 缺失或版本不对
- CA Index 不匹配
- Issuer / ICC 证书解析错误
- 哈希输入数据顺序错误
- CDA 响应模板理解错误

### 5. 工程修复建议

- 把 ODA 实现拆成更细的错误码
- 日志中明确打印命中的 RID / Index / CAPK 标识
- 建立 ODA 失败样本回放能力

## 案例 3：CVM 行为和用户看到的不一致

### 1. 现象

- 用户说自己输入过 PIN
- 但主机侧看到的是 No CVM
- 或者终端 UI 显示无需验证，但 `9F34` 显示另一种 CVM

### 2. 关键日志

```text
+[CVM] result=NO_CVM
+[TLV] 9F34=1E0300
+```

### 3. 正确分析路径

优先相信结构化结果，再反查 UI 和设备层：

1. 看 CVM List 和终端能力
2. 看金额、限额和品牌条件
3. 看 `9F34` 最终值
4. 再看 UI 层和 PIN Pad 行为是否同步

### 4. 高概率根因

- UI 与内核状态不同步
- CVM 条件判断错误
- PIN bypass / 超时 / 用户取消未正确映射
- 终端能力参数错误

### 5. 工程修复建议

- `9F34` 统一由内核生成，业务层只消费结果
- CVM 结果分清成功、失败、不可执行、取消、超时

## 案例 4：主机说 ARQC 校验失败

### 1. 现象

- 终端看起来交易流程正常
- 但主机返回 ARQC verify failed 或等价错误

### 2. 关键日志

```text
+[AC1] CID=ARQC ATC=0012 AC=...
+[ISO8583] F55=82...95...9F26...9F27...9F36...9F37...
+[HOST] arqc verify failed
+```

### 3. 正确分析路径

优先检查：

1. Field 55 是否漏标签
2. `9F26`、`9F36`、`9F37`、`95`、`9F10` 是否正确
3. UN 是否合规
4. IAD 是否来自正确来源
5. 编码与长度是否符合主机约定

### 4. 高概率根因

- Field 55 漏标签
- 标签值编码错误
- 把十六进制串当文本送主机
- UN 质量不合格
- IAD/CVR 处理错误

### 5. 工程修复建议

- 建立联机前完整标签快照
- 建立主机字段映射单元测试
- 不同品牌分开实现 host mapping

## 案例 5：主机批准，但卡片最终拒绝

### 1. 现象

- 主机授权响应码是 `00`
- 终端却最终显示交易失败
- 二次 Generate AC 返回 AAC

### 2. 关键日志

```text
+[HOST] response=00 arpc=... script72=...
+[SCRIPT] result=6985
+[AC2] CID=AAC
+[OUTCOME] DECLINED
+```

### 3. 正确分析路径

这时不要只盯主机结果，应看：

1. ARPC 是否回传并被正确消费
2. Script 71 / 72 是否按正确时机执行
3. Script APDU 是否成功
4. 二次 Generate AC 输入是否完整
5. TVR 是否记录 issuer/script 异常

### 4. 高概率根因

- ARPC 没正确带回卡
- Script 时机错误
- Script 执行失败
- 二次 Generate AC 输入错误
- 主机批准但卡片侧状态不接受最终完成

### 5. 工程修复建议

- 将联机后处理独立成明确阶段
- 对 Script 和 AC2 单独做日志与回放
- 不要把“主机批准”直接映射成“最终成功”

## 案例 6：非接交易频繁要求插卡

### 1. 现象

- 用户刷非接卡
- 终端经常提示改为插卡
- 交易并没有进入预期的 contactless kernel 深流程

### 2. 关键日志

```text
+[RF] card detected
+[PPSE] candidates=...
+[ENTRY] route result=TRY_ANOTHER_INTERFACE
+```

### 3. 正确分析路径

优先看：

1. PPSE 是否返回候选应用
2. Entry Point 是否路由到预期 kernel
3. Reader capability / TTQ / limit 参数是否合理
4. 是否命中品牌或终端的 fallback 条件

### 4. 高概率根因

- Entry Point 路由错误
- 非接 Reader 参数错误
- TTQ 设置不匹配
- CVM limit / reader limit 配置不合理
- 候选应用与品牌 profile 映射错误

### 5. 工程修复建议

- 路由日志结构化输出
- Entry Point、Reader 参数和品牌 profile 联合调试
- 将 fallback 原因枚举化，而不是只打一条模糊提示

## 案例 7：同一张卡在不同终端表现不一致

### 1. 现象

- A 终端交易正常
- B 终端同卡同场景失败
- 日志看上去主线差不多，但结果不同

### 2. 正确分析路径

优先比较：

1. 参数版本是否一致
2. AID / CAPK / TAC / Reader 参数是否一致
3. 日志中的 TVR / TSI / 9F34 / CID 是否一致
4. Field 55 是否一致

### 3. 高概率根因

- CAPK 表版本不一致
- Floor Limit / TAC / Reader limit 配置漂移
- 某台终端代码版本不同
- 日志开关不同导致误判

### 4. 工程修复建议

- 日志打印参数版本
- 建立“问题发生时生效参数快照”能力
- 终端环境、代码版本、参数版本统一纳管

## 2. 案例库的正确用法

建议不要把案例库当成“背答案”，而要当成训练模板：

- 先看现象
- 自己先判断优先排查方向
- 再对照正确分析路径
- 最后总结这类问题属于哪一类根因

## 3. 建议的根因分类

你可以把大多数 EMV 问题归入下面 6 类：

1. 输入组包问题
2. TLV / DOL / APDU 解析问题
3. 风险与位图问题
4. 联机映射问题
5. 联机后处理问题
6. 参数 / 品牌 / 环境差异问题
