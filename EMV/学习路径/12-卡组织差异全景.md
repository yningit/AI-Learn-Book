# 12 - 卡组织差异全景

## 1. 本文的定位

这篇文档不是要逐条替代各卡组织的正式规范，而是试图从“参考书 + 工程实践”的视角，给出一张更完整的卡组织差异地图。

适用范围包括：

- Visa
- Mastercard
- UnionPay
- American Express（Amex）
- Discover / Diners Club
- RuPay
- Pure
- JCB

它的目标是帮助你理解：

- 哪些是 EMV 的通用骨架
- 哪些是各卡组织在工程实现、测试和联调中的典型差异点
- 为什么项目上必须把 scheme 逻辑做成可治理、可隔离、可测试的插件体系

> 说明：不同卡组织规范版本、地区规则、认证要求会持续演进；本文强调“差异类型”和“工程落点”，不宣称替代官方规范。

## 2. 先建立一个总原则

所有卡组织都建立在 EMV 通用思想之上，但它们会在以下方面引入自己的规则：

1. 应用入口与路由
2. 非接触式行为与 Outcome 模型
3. CVM、限额与 reader 参数
4. 发卡行数据对象解释
5. 联机字段、IAD / CVR 结构和认证要求
6. 测试案例与认证关注点

所以工程上最重要的能力不是“死记某一家”，而是：

- 先掌握通用骨架
- 再识别差异落点
- 最后把差异隔离为 scheme profile

## 3. 各卡组织差异全景表

| 卡组织 | 常见工程关注点 | 最常落地的差异模块 |
| --- | --- | --- |
| Visa | 应用/品牌 profile 清晰隔离、联机映射和非接参数协同 | host mapping、reader profile、brand profile |
| Mastercard | 非接 outcome 和 reader 交互模型、测试 case 细节 | outcome engine、entry point、scheme rules |
| UnionPay | 本地受理环境、参数组合和主机配合场景 | terminal config、host mapping、scheme profile |
| Amex | 品牌专有对象、联机与风险路径细节 | object parser、host mapping、scheme rules |
| Discover / Diners | 受理网络与品牌路径兼容性、对象解释差异 | routing、host mapping、scheme profile |
| RuPay | 本地化受理规则、参数与主机协同 | config profile、host mapping、test profile |
| Pure | 本地或区域化 profile、非接/接触路径约束 | AID profile、terminal config、scheme rules |
| JCB | 品牌 profile、限额与 host 协作细节 | host mapping、CVM/risk rules、scheme profile |

## 4. 应用入口与路由差异

### 4.1 通用部分

无论哪家卡组织，入口都绕不开：

- 目录选择或候选应用发现
- 最终 AID 确定
- 接触 / 非接路径进入
- kernel 或 profile 接管

### 4.2 差异通常体现在哪

- 非接场景下是否强依赖 Entry Point / Kernel ID
- 候选应用冲突时优先级与选择行为
- 回退到接触式或其他接口的条件
- 品牌 profile 如何与 AID 集合绑定

### 4.3 工程建议

- 路由逻辑必须独立于品牌内核实现
- 用 `SchemeProfile`、`KernelCandidate`、`RouteDecision` 这类模型抽象差异
- 统一保留路由日志，记录“为什么走到这个 scheme”

## 5. 关键对象与数据结构差异

### 5.1 通用对象

这些对象在大多数卡组织里都能看到：

- AID
- AIP
- AFL
- TVR
- TSI
- CID
- ATC
- `9F34`

### 5.2 差异最常出现在

- IAD
- CVR
- scheme 私有数据对象
- 非接 outcome 或 reader action 相关对象
- 部分主机侧解释字段

### 5.3 为什么这很重要

因为项目中最常见的错误之一，就是把某一家卡组织的 IAD/CVR 解释方式，误用到其他卡组织上。

### 5.4 工程建议

- 通用 Tag 统一建模
- scheme 私有对象统一放在品牌解释层
- 日志中区分“通用对象”和“scheme 私有对象”

## 6. CVM、限额与 reader 参数差异

### 6.1 通用目标

所有卡组织最终都要回答：

- 这笔交易需不需要做 CVM
- 该做哪种 CVM
- 小额场景是否允许 No CVM
- 是否应联机、回退或终止

### 6.2 差异常见在

- CVM limit 的实际使用方式
- reader limit / contactless limit 的配置约束
- PIN、签名、No CVM 的偏好顺序
- fallback 与 outcome 条件

### 6.3 各卡组织的常见工程观察

#### Visa

- 常需要把 reader 参数、品牌 profile 和联机字段一起看
- 参数不一致时，很容易在非接和联机路径表现异常

#### Mastercard

- 非接 outcome 行为和 reader 指令抽象通常更值得关注
- 测试 case 往往会较细地覆盖用户体验和 reader 行为

#### UnionPay

- 本地受理参数和终端主机协同常更突出
- 不同商户/地区配置组合对结果影响明显

#### Amex

- 品牌私有数据对象和 host 解释路径值得重点隔离
- 认证和联调时不要假设与其他国际品牌字段解释完全一致

#### Discover / Diners

- 受理网络与品牌 profile 的匹配需要特别小心
- routing、host mapping 和 acceptance profile 的关系更值得关注

#### RuPay

- 本地受理环境中的参数与 host 协作通常是关键
- 配置治理质量直接影响现场一致性

#### Pure

- 常见于特定地区或项目 profile，配置和路由设计要足够灵活
- 不要把其视为“某个国际品牌的简化版”去硬套逻辑

#### JCB

- 品牌 profile、联机字段和终端参数配合值得单独测试
- 主机和终端的字段解释一致性非常关键

## 7. 联机、Field 55 与发卡行数据差异

### 7.1 通用部分

所有卡组织都离不开：

- AC 结果上送
- Field 55 组装
- ARQC 上传
- ARPC / Script 回传
- 二次 AC 闭环

### 7.2 差异最常落在哪

- Field 55 必传标签集合
- 标签排序和编码约束
- IAD / CVR 的结构和主机解释
- 授权后脚本处理细节

### 7.3 工程建议

- 建立 scheme-specific host mapping 层
- 所有品牌都保存联机前的标签快照
- 不同卡组织使用不同的字段解释器和测试样例

## 8. Outcome、UI 和 reader 交互差异

### 8.1 为什么这一点常被忽视

很多团队只关注“是否交易成功”，但非接特别容易受到：

- reader 行为
- UI 提示
- fallback 指令
- outcome 抽象模型

的影响。

### 8.2 工程建议

- 把品牌 outcome 映射到统一的终端 Outcome 对象
- 业务层只消费统一对象，不直接消费 scheme 私有结果
- 对用户提示、蜂鸣、灯光、继续/终止/插卡提示做结构化建模

## 9. 测试与认证差异

### 9.1 通用事实

所有卡组织都会通过测试或认证来验证：

- 交易主流程是否符合要求
- 风险与 CVM 处理是否正确
- 联机与后处理是否正确
- 品牌专有行为是否正确

### 9.2 差异常见在

- case 的预期日志与对象值
- outcome 行为要求
- Field 55 与 host 配合要求
- 非接参数和 reader 行为要求

### 9.3 工程建议

- 为每个卡组织维护独立 test profile
- 为每个卡组织维护关键日志检查点
- 不同卡组织的认证 case 结果要独立归档

## 10. 一个建议的代码组织方式

```text
Common EMV Engine
  -> Entry Point / Route Layer
      -> Scheme Profile Registry
          -> Visa Profile
          -> Mastercard Profile
          -> UnionPay Profile
          -> Amex Profile
          -> DiscoverDiners Profile
          -> RuPay Profile
          -> Pure Profile
          -> JCB Profile
```

每个 profile 至少可包含：

- AID / route profile
- object parser
- CVM / risk / outcome rules
- host mapping
- test profile
- config schema

## 11. 参考书式使用建议

如果你把这篇文档当成参考书章节，建议这样使用：

### 当你遇到品牌联调问题时

先看：

- 这属于哪个卡组织
- 差异落在入口、对象、联机还是 outcome
- 是代码问题还是参数/profile 问题

### 当你设计架构时

先问：

- 这个能力是通用的，还是 scheme-specific 的
- 应该落在 Common Engine 还是 Scheme Profile

### 当你准备测试时

先问：

- 哪些 case 是通用回归
- 哪些 case 是品牌专属验证

## 12. 初学者最容易误解的地方

### 12.1 以为“品牌差异”只是多几个 AID

不对。真正困难的是：

- 对象解释差异
- reader 参数差异
- outcome 差异
- host mapping 差异
- 测试认证差异

### 12.2 以为只要抽象一个接口就够了

也不对。仅有接口抽象不够，还要有：

- 参数隔离
- 日志隔离
- host mapping 隔离
- test profile 隔离

### 12.3 以为小卡组织就可以“顺带兼容”

风险很大。像 Amex、Discover/Diners、RuPay、Pure、JCB 这类组织，虽然在某些项目中交易量不一定最大，但在实现和联调上同样需要明确 profile 和验证路径。

读完后，你应该能：

- 知道不同卡组织的差异大致落在哪些模块
- 知道为什么工程实现必须做 scheme profile 隔离
- 在遇到 Amex、Discover/Diners、RuPay、Pure、JCB 等组织需求时，不会只从 Visa/Mastercard 的思路去硬套

下面给出一个更适合工程落地时快速翻阅的速查表：

| Visa | 非接 profile、CVM 和 outcome 隔离 | Field 55、IAD 解释、联机一致性 | 非接行为、outcome、联机后处理 |
| Mastercard | 非接 kernel 行为、reader 参数、outcome | ARQC/ARPC、IAD/CVR、主机消费一致性 | case 细节、异常路径和结果呈现 |
| UnionPay | 参数组合、本地受理场景、主机协同 | host mapping、本地字段要求、环境一致性 | 参数正确性、联机路径、场景覆盖 |
| Amex | 独立 scheme profile、专有对象解释 | 私有对象映射、联机字段要求 | 品牌专项 case 与参数隔离 |

| Pure | 项目型 profile、定制参数、独立治理 | 定制 mapping 与环境控制 | 项目专属 case 和样机一致性 |
| JCB | 品牌 profile 与参数协同 | 主机字段与品牌对象映射 | 品牌专项 case 和联机验证 |
