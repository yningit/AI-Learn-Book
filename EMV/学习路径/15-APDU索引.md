# 15 - APDU 索引

这份索引整理 EMV 交易中最常见、最值得优先掌握的 APDU 命令，适合作为“参考书附录”快速查阅。

> 说明：这里侧重工程用途、阅读日志和定位问题时的理解，不替代正式 APDU 规范细节。

## 一、APDU 的基本结构

一个命令 APDU 常见由以下部分组成：

- CLA：命令类别
- INS：指令码
- P1 / P2：参数
- Lc：命令数据长度
- Data：命令数据
- Le：期望响应长度

一个响应 APDU 常见由以下部分组成：

- Response Data：响应数据
- SW1SW2：状态字

## 二、最常见的 EMV APDU 命令总表

| 命令 | 常见用途 | 常见阶段 | 工程关注点 |
| --- | --- | --- | --- |
| `SELECT` | 选择目录或应用 | Application Selection | 是否选中正确 AID / PSE / PPSE |
| `GET PROCESSING OPTIONS` | 获取 AIP / AFL | GPO | PDOL 是否组装正确 |
| `READ RECORD` | 读取应用记录 | Read Record | 是否读全关键对象 |
| `VERIFY` | 验证 PIN | CVM | PIN 路径和结果映射是否正确 |
| `GENERATE AC` | 生成 AC | AC1 / AC2 | CDOL 组装和 CID 解析是否正确 |
| `EXTERNAL AUTHENTICATE` | 发卡行认证相关 | 联机后处理 | ARPC / issuer auth 路径是否正确 |
| `GET DATA` | 获取数据对象 | 部分实现 / 品牌扩展 | 品牌私有或补充对象读取 |
| Script APDU | 执行发卡行脚本 | 联机后处理 | 时机、顺序和 SW1SW2 是否正确 |

## 三、`SELECT`

### 1. 作用

用于选择：

- PSE / PPSE
- 某个 AID
- 某个目录或应用文件

### 2. 工程意义

`SELECT` 的核心价值是确定“由谁来接管交易”。

### 3. 日志里看什么

- 选择的是 PSE / PPSE 还是 AID
- 最终成功的是哪个 AID
- 是否返回 `9000`
- 是否带回 `84`、`50`、`87`、`9F38` 等关键对象

### 4. 常见问题

- 候选应用发现不完整
- 选中错误 AID
- 目录存在但应用选择策略错误

## 四、`GET PROCESSING OPTIONS`

### 1. 作用

用于启动应用处理流程，卡片通常返回：

- `82` AIP
- `94` AFL

### 2. 工程意义

GPO 是交易进入执行主线的关键分叉点。

### 3. 日志里看什么

- PDOL 输入是否正确
- AIP 是否合理
- AFL 是否合理
- 响应模板是否符合预期

### 4. 常见问题

- PDOL 长度错误
- 编码错误
- 缺字段
- 非接少了 TTQ 等必要对象

## 五、`READ RECORD`

### 1. 作用

按照 AFL 指示，逐条读取卡片应用记录。

### 2. 工程意义

这是卡片主要业务对象进入终端 TLV 仓库的阶段。

### 3. 日志里看什么

- SFI 和记录号是否符合 AFL
- 是否成功读取 PAN、有效期、CVM List、CDOL1、CDOL2、ODA 对象
- 是“记录不存在”还是“读取失败”

### 4. 常见问题

- 没按 AFL 顺序读取
- 关键记录未读到
- 读到了但 TLV 解析错误

## 六、`VERIFY`

### 1. 作用

用于 Offline PIN 验证。

### 2. 工程意义

它影响 CVM 结果、`9F34`、TVR 以及后续动作路径。

### 3. 日志里看什么

- 是否真的进入 Offline PIN 路径
- PIN 输入是否成功
- 卡片返回状态字是什么
- 最终 `9F34` 是否符合预期

### 4. 常见问题

- UI 与内核状态不同步
- PIN bypass / timeout 未正确映射
- 终端能力与卡要求不匹配

## 七、`GENERATE AC`

### 1. 作用

这是卡片对交易进行密码学决策的核心命令。

### 2. 工程意义

`GENERATE AC` 通常要看两次：

- 第一次：AC1，决定 AAC / TC / ARQC
- 第二次：AC2，在联机后决定最终 TC / AAC

### 3. 日志里看什么

- CDOL 输入是否正确
- 返回的 CID 是什么
- 是否带回 `9F26`、`9F27`、`9F36`、`9F10`
- AC1 和 AC2 是否与主机结果一致

### 4. 常见问题

- CDOL 数据长度错误
- CID 解释错误
- AC2 输入不完整
- 主机批准但 AC2 返回 AAC

## 八、`EXTERNAL AUTHENTICATE`

### 1. 作用

常用于把发卡行认证数据带回卡片，用于 issuer authentication 路径。

### 2. 工程意义

不是所有路径都会用到，但一旦使用，往往和 ARPC / issuer authentication 正确性高度相关。

### 3. 日志里看什么

- 是否在正确时机调用
- 输入是否包含正确的 issuer auth data
- 卡片状态字是否成功

## 九、Issuer Script APDU

### 1. 作用

执行主机返回的脚本命令。

### 2. 工程意义

脚本是否成功，常常直接影响 TVR、日志和最终交易结果。

### 3. 日志里看什么

- `71` / `72` 的执行时机
- 每条脚本 APDU 是否成功
- SW1SW2 是什么
- 是否影响 AC2 结果

### 4. 常见问题

- 前后脚本时机颠倒
- 脚本失败未记录
- 脚本失败后未正确更新 TVR 或结果

## 十、常见 SW1SW2 理解建议

这里不追求完整状态字字典，而给出工程上常见的理解方式：

| 状态字 | 常见理解 | 排查方向 |
| --- | --- | --- |
| `9000` | 成功 | 继续看响应数据是否合理 |
| `6A80` | 常见为数据字段或格式错误 | 先查 DOL / TLV / 输入编码 |
| `6985` | 常见为条件不满足 | 先查流程时机、对象是否齐全 |
| 其他非成功状态 | 需结合命令和上下文判断 | 不要只看状态字，必须看阶段与输入 |

## 十一、一个建议的 APDU 排查顺序

拿到 APDU 日志后，建议固定按这个顺序看：

1. 当前是哪条命令
2. 这条命令属于交易哪个阶段
3. 输入数据由哪些对象拼出来
4. 响应里最关键的数据对象是什么
5. 状态字代表“输入错”、“条件不满足”还是“品牌/卡不支持”
6. 这条命令的结果如何影响后续 TVR / TSI / CID / Field 55

## 十二、初学者最容易踩的坑

1. 只看状态字，不看响应数据。  
2. 只看某一条 APDU，不看它所处的交易阶段。  
3. 看到 `9000` 就以为没问题，忽略了返回对象本身不完整。  
4. 不把 APDU 和日志中的 TLV、TVR、CID 结果联动起来看。
