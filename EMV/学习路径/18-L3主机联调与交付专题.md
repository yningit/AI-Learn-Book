# 18 - L3 主机联调与交付专题

## 1. 为什么还要讲 L3

很多团队把 EMV 工作只看到 L2 或 Kernel 实现层，但真正项目落地时，经常卡在另一类问题上：

- 内核看起来没问题，但主机联调不过
- Field 55 能发，但主机解释不一致
- 主机批准了，业务系统没有正确收口
- 冲正、异常恢复、批结算、回执、日志归档没打通

这类问题通常不再只是 L2 问题，而更接近：

- 终端业务集成
- 主机字段协同
- 异常恢复与清算准备
- 项目交付质量控制

这也是为什么需要单独理解 L3 / 主机联调 / 项目交付层。

## 2. L1、L2、L3 的工程理解

### 2.1 L1

更多偏：

- 读卡器物理能力
- 接触 / 非接协议链路
- 底层时序和通信能力

### 2.2 L2

更多偏：

- EMV Kernel 流程实现
- ODA / CVM / Risk / AC / Script 行为
- 品牌规则执行与测试认证

### 2.3 L3

更多偏：

- POS 应用与 Kernel 集成
- 主机联调与报文协同
- 冲正、超时、异常恢复
- 业务流程、凭条、批结算、验收与上线准备

### 2.4 一个简化理解

- L1：能不能稳定读卡
- L2：内核是不是按规范工作
- L3：项目是不是能真实上线并稳定运营

## 3. L3 最常见的工作范围

### 3.1 终端业务集成

- 金额输入
- 交易类型选择
- 取消 / 超时 / 重试
- 电子签名 / 打印 / 回执
- 联机失败后的用户提示和业务收口

### 3.2 主机联调

- ISO8583 报文映射
- Field 55 一致性
- 授权响应消费
- ARPC / Script 回传
- 冲正和重发策略

### 3.3 项目交付

- 参数发布
- 证书与环境准备
- 版本封版
- 测试结论与上线门槛
- 运行监控与回滚准备

## 4. 主机联调时最重要的对象

在 L3 阶段，下面这些对象最常成为联调焦点：

| 类别 | 关键对象 |
| --- | --- |
| 终端交易输入 | Amount、Currency、Transaction Type、Sequence Counter |
| EMV 上送对象 | AIP、TVR、TSI、CVM Results、CID、ATC、IAD、UN |
| 主机响应对象 | ARC、Authorization Code、ARPC、Issuer Script |
| 业务控制对象 | 终端流水号、批次号、参考号、冲正标识、超时状态 |

## 5. Field 55 在 L3 中为什么这么关键

L2 关注 Field 55 是否按规则组装，L3 还要进一步关注：

- 主机是否按同样的语义解释这些标签
- 主机和终端是否对标签来源达成一致
- 标签缺失时主机是返回失败、降级还是走备用逻辑

### 5.1 L3 里常见的 Field 55 问题

- 终端认为标签正确，但主机要求的顺序不同
- 主机需要某个标签，但终端没上送
- 标签值来源不一致，如 IAD/TVR/UN 使用错误值
- 把十六进制串当成文本送到主机

### 5.2 工程建议

- 保存“联机前 EMV 标签快照”
- 维护“EMV Context -> ISO8583 字段映射表”
- 对每个品牌单独维护 host mapping profile

## 6. 主机联调中最常见的 5 类问题

### 6.1 ARQC 校验失败

优先排查：

- `9F26`、`9F36`、`9F37`、`95`、`9F10`
- Field 55 编码
- IAD 来源
- 主机密钥 / 配置版本

### 6.2 主机批准但终端失败

优先排查：

- ARPC 是否正确消费
- Script 是否执行成功
- 二次 Generate AC 是否返回 AAC
- 业务层是否误读 Outcome

### 6.3 联机超时或重发导致状态不一致

优先排查：

- 超时后的交易状态持久化
- 冲正是否触发
- 业务层和内核是否都清楚“这笔交易是否已经发出”

### 6.4 终端显示成功，但主机未落账

优先排查：

- 主机返回消费是否被真正确认
- 是否存在中间链路超时
- POS 应用和 Host Adapter 是否把结果落库

### 6.5 不同环境表现不一致

优先排查：

- 测试 / 生产参数差异
- CAPK / AID / Host 配置版本差异
- 主机字段开关和品牌 profile 差异

## 7. 冲正与异常恢复为什么必须纳入交付范围

如果系统只会“正常交易”，但不会处理异常，那么项目上线后问题会很快暴露。

L3 必须关注：

- 联机超时后是否需要冲正
- 交易发出但结果未知时是否可恢复
- 打印 / UI 失败是否影响交易最终状态
- 脚本和 AC2 失败后是否能正确保留现场数据

## 8. 一个建议的 L3 联调顺序

建议不要一上来就做“大联调”，更好的顺序是：

1. 单机日志检查：Kernel 输出是否完整
2. 本地报文检查：Field 55 和关键域是否正确
3. 主机单链路联调：授权请求 / 响应闭环
4. 异常联调：超时、拒绝、冲正、重发
5. 业务联调：回执、批次、流水、对账、后台状态

## 9. 项目交付时最容易遗漏的点

### 9.1 只测成功路径

风险：上线后超时、拒绝、脚本失败、非接回退等场景失控。

### 9.2 内核日志完善，但业务日志缺失

风险：现场问题只能看到 APDU，看不到业务状态变化和冲正流程。

### 9.3 参数已下载，但无法确认生效版本

风险：问题出现时无法知道终端到底跑的是哪一版参数。

### 9.4 Host mapping 文档不完整

风险：主机联调时双方对字段解释不一致。

## 10. 对团队协作的建议

### 开发团队

- 明确 Kernel 与 POS App 的接口边界
- 提供联机前后快照
- 对超时、取消、重试建立统一状态机

### 主机团队

- 明确 Field 55 与关键域要求
- 明确 ARPC / Script 返回规则
- 提供错误码与失败场景解释

### 测试团队

- 不仅验证成功授权，还要覆盖拒绝、超时、冲正、重发
- 验证 UI / 日志 / 主机记录是否一致

## 11. 一个建议的交付结论模板

每次项目封版或阶段交付，建议输出至少包含：

- 支持的交易类型
- 已通过的品牌 / 接口 / 场景
- 已验证的联机路径
- 已验证的异常路径
- 当前参数版本
- 当前证书 / CAPK 版本
- 已知限制与待跟进项

## 12. 这一册学完后，你应该达到什么程度

读完后，你应该能：

- 理解为什么 L3 不只是“再跑一遍交易”
- 知道主机联调、冲正、异常恢复和交付管理的重要性
- 理解从 L2 到真实项目上线之间还差哪些工程能力
