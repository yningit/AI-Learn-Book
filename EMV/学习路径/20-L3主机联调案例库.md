## 案例 1：终端送了 Field 55，但主机提示缺少关键标签

### 1. 现象

- 终端显示已成功发起联机授权
- 报文中确实存在 Field 55
- 主机返回“缺少 EMV data”或等价错误

### 2. 优先看什么

1. Field 55 是否真的包含主机要求的标签集合
2. 是否把二进制 TLV 转成了文本串
3. 标签长度是否与实际值一致
4. 品牌 profile 是否用了错误模板

### 3. 高频根因

- `9F26`、`9F27`、`9F10`、`9F37`、`95`、`9A`、`9C`、`9F02` 有缺失
- Tag 顺序或拼接方式不符合主机约定
- 中间层把十六进制字节流当成字符串处理
- 接触 / 非接用了同一份不完整映射

### 4. 工程建议

- 保留联机前原始 TLV 快照
- 保留 ISO8583 序列化前后的 Field 55 对照
- 每个品牌单独维护字段映射用例

## 案例 2：主机提示 ARQC 校验失败，但内核日志看起来正常

### 1. 现象

- `GENERATE AC 1` 返回 ARQC
- 内核日志没有明显错误
- 主机返回 `ARQC verify failed`

### 2. 优先看什么

1. `9F26`、`9F36`、`9F37`、`95`、`9F10` 的上送值
2. UN 是否来自正确随机源
3. IAD 是否直接来自卡而非业务层重组
4. 金额、币种、日期、交易类型是否和主机参与验签的数据一致

### 3. 高频根因

- UN 质量不达标或重复
- `9F10` 被截断或错误编码
- Amount / Currency / Date 字段和主机解释不一致
- 测试主机和终端所用密钥版本不一致

### 4. 工程建议

- 对 ARQC 相关标签建立固定审计集合
- 把主机验签输入抽成一张联调对照表
- 区分“内核生成正确”和“主机使用正确”这两个维度

## 案例 3：主机批准了，但终端最终显示拒绝

### 1. 现象

- 主机响应码为 `00`
- 终端最后显示 Declined 或失败
- 业务团队误以为“主机已经批准，终端一定成功”

### 2. 优先看什么

1. 响应中的 ARPC 是否被正确消费
2. Script 71 / 72 是否存在且是否执行成功
3. 二次 `GENERATE AC` 是否返回 AAC
4. 最终 Outcome 是否被业务层覆盖

### 3. 高频根因

- ARPC 被截断或映射到错误字段
- Script 执行失败后没有被正确呈现
- AC2 输入缺失导致卡最终 AAC
- POS App 用主机响应码覆盖了真实卡片结果

### 4. 工程建议

- 把联机后处理拆成独立阶段日志
- 明确区分“Host approved”和“Card accepted”
- 业务层只能消费内核最终结果，不能自行改判

## 案例 4：终端显示成功，但主机没有入账记录

### 1. 现象

- 收银员看到交易成功
- 终端打印小票成功
- 主机或后台查不到该笔交易

### 2. 优先看什么

1. 授权请求是否真的发出成功
2. 响应返回前是否发生链路超时
3. POS 应用是否把“发送成功”误当成“交易成功”
4. 是否存在异步落库失败

### 3. 高频根因

- 中间件发包成功但未收到正式授权响应
- 业务层把 socket 成功当授权成功
- 主机已响应但终端本地保存失败
- 打印 / UI 流程提前掩盖了真实交易状态

### 4. 工程建议

- 交易状态机至少区分：已发送、待确认、已授权、已冲正、未知
- 结果展示必须依赖最终状态而不是网络动作
- 保留终端流水号与主机参考号映射

---

## 案例 5：联机超时后发生重复扣款争议

### 1. 现象

- 第一次交易超时
- 收银员或用户再次发起交易
- 之后主机发现两笔都成功或一笔待确认、一笔成功

### 2. 优先看什么

1. 超时后终端是否进入待确认状态
2. 是否触发冲正或查询机制
3. 第二次交易是否复用了错误的流水或上下文
4. 主机是否支持幂等或重复交易识别

### 3. 高频根因

- 超时交易未做状态冻结
- 没有冲正 / 查询补偿机制
- 操作员可以立即重试且没有提示风险
- 终端和主机对“未知状态交易”定义不同

### 4. 工程建议

- 为超时交易定义明确的中间状态
- 引入冲正、查询或后台补偿流程
- UI 上把“超时”与“失败”明确区分

---

## 案例 6：ARPC 返回了，但脚本始终不执行

### 1. 现象

- 主机响应里带了 ARPC 和 Issuer Script
- 终端日志显示联机成功
- 没有任何 Script 执行记录

### 2. 优先看什么

1. Host Adapter 是否正确解析脚本字段
2. 脚本是否被放入内核期望的数据结构
3. 联机后处理阶段是否真的被触发
4. 是否只有某个品牌 profile 缺了脚本映射

### 3. 高频根因

- 响应脚本被业务层丢弃
- Script 标签封装格式和内核接口约定不一致
- 联机成功后流程直接提前结束
- 品牌专有字段未正确路由

### 4. 工程建议

- 对脚本建立“收到 / 注入 / 执行 / 结果”四段日志
- Host Adapter 与 Kernel 间用稳定 DTO，避免裸字段拼接

## 案例 7：主机和终端对响应码理解不一致

### 1. 现象

- 主机返回某个非 `00` 的业务码
- 终端 UI 显示的文案与主机预期不一致
- 现场难以判断是否需要重试、冲正或换卡

### 2. 优先看什么

1. Host response code 与业务错误码映射表
2. 内核结果、主机结果、界面提示之间的优先级
3. 是否不同品牌共用了同一套文案表

### 3. 高频根因

- 主机码语义未文档化
- POS App 直接用通用“交易失败”兜底
- 不同主机环境返回码含义不同

### 4. 工程建议

- 建立按主机 / 品牌维度的响应码字典
- 将“是否可重试”“是否应冲正”“是否需换卡”做成结构化标识

## 案例 8：测试环境正常，生产环境异常

### 1. 现象

- SIT / UAT 全部通过
- 切到生产环境后同卡同流程失败
- 团队误判为代码回归

### 2. 优先看什么

1. AID / CAPK / TAC / limit 参数版本差异
2. 主机地址、证书、白名单、路由差异
3. 品牌 profile 是否在生产环境单独配置
4. 是否存在测试卡与真卡行为差异

### 3. 高频根因

- 参数包未同步
- CAPK 环境混用
- 主机环境开关不同
- 生产链路增加了网关或风控校验

### 4. 工程建议

- 任何联调结论都必须带环境版本号
- 保留参数差异对比表和主机环境矩阵

## 9. 一套通用的 L3 排查顺序

无论碰到什么联调问题，都建议按下面顺序推进：

1. 终端输入是否正确
2. 内核阶段输出是否正确
3. Field 55 / ISO8583 映射是否正确
4. 主机响应是否完整返回
5. 联机后处理是否完整执行
6. 业务结果、日志结果、主机结果是否一致

## 10. L3 案例和 L2 问题的边界

可以用一个简单判断帮助自己快速分层：

- 如果问题发生在 APDU、ODA、CVM、Risk、AC 行为本身，更像 L2
- 如果问题发生在报文映射、主机协同、异常恢复、结果收口，更像 L3
- 如果两边都有迹象，就先固定卡片侧结果，再追业务链路

## 11. 你应该沉淀哪些联调资产

每做完一轮项目，建议至少沉淀：

- Field 55 标签对照表
- 主机响应码字典
- 常见问题现象 -> 根因速查表
- 典型超时 / 冲正 / 脚本失败回放日志
- 按品牌拆分的 host mapping 配置说明

这样下一次项目的 L3 联调效率会明显提高。
