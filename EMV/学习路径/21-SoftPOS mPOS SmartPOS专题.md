# 21 - SoftPOS / mPOS / SmartPOS 专题

## 1. 为什么要单独讲终端形态

很多团队学 EMV 时默认脑海里的终端是“传统 POS 机”，但真实项目里常见的终端形态已经很多：

- 传统 POS
- mPOS
- SmartPOS
- SoftPOS

它们并不是“换个壳”这么简单。

从工程角度看，不同形态会影响：

- 读卡方式和外设架构
- PIN 输入和安全域设计
- EMV Kernel 的部署位置
- 认证测试范围
- 日志、升级、参数下载和运维方式

所以，如果要把这套资料作为工程参考书，这一层最好单独说明。

## 2. 四类终端的直观理解

| 形态 | 简化理解 | 典型设备形态 |
| --- | --- | --- |
| 传统 POS | 一体式支付终端 | 专用收单终端 |
| mPOS | 小读卡器 + 手机 / 平板 | 蓝牙读卡头 + App |
| SmartPOS | 智能终端内置支付能力 | Android 智能 POS |
| SoftPOS | 通用手机直接受理支付 | Android 手机 Tap to Pay |

## 3. 传统 POS

### 3.1 特点

- 设备形态稳定
- 读卡器、PIN Pad、打印机通常一体化
- 安全边界清晰
- EMV 方案成熟

### 3.2 工程含义

- Kernel、读卡器驱动、PIN 模块、打印模块通常在同设备内协同
- 外设和支付软件边界相对固定
- 适合强调稳定性和认证闭环

### 3.3 常见挑战

- 系统老旧、升级节奏慢
- 多型号适配成本高
- UI 和业务灵活性相对有限

## 4. mPOS

### 4.1 特点

- 手机 / 平板承担业务 App
- 小型读卡器承担刷卡 / 插卡 / PIN 输入能力
- 两个设备通过蓝牙或音频等方式协同

### 4.2 工程含义

- Kernel 可能在外设侧，也可能部分在移动端侧
- 终端状态和读卡器状态需要跨设备同步
- 日志往往分散在读卡器、SDK、移动 App 三侧

### 4.3 常见挑战

- 蓝牙链路不稳定
- 设备配对与版本兼容问题
- UI 事件和 EMV 阶段不同步
- 移动端 App 更新频繁，联调面更大

## 5. SmartPOS

### 5.1 特点

- Android 系统
- 集成接触、非接、扫码、打印、摄像头等能力
- 业务 App 和支付能力可深度集成

### 5.2 工程含义

- 通常可以在同一设备里完成 POS App、Kernel、外设控制和主机联调
- 与传统 POS 相比，应用层扩展能力更强
- 更容易引入复杂业务流程和第三方组件

### 5.3 常见挑战

- 系统权限、进程模型、厂商 SDK 差异大
- 不同机型之间驱动和安全实现存在差异
- 支付核心能力与普通 Android 能力耦合后，稳定性治理更重要

## 6. SoftPOS

### 6.1 特点

- 使用通用 Android 手机直接做非接受理
- 通常依赖 NFC
- 更强调软件能力和远程交付能力

### 6.2 工程含义

- 没有传统独立 PIN Pad 和专用安全外设的假设
- 对设备能力、系统版本、NFC 行为、TEE / 安全模块要求更敏感
- 参数治理、风控、设备完整性检测会更重要

### 6.3 常见挑战

- 设备碎片化严重
- NFC 行为在不同机型上差异明显
- 某些支付路径和金额场景可能受限
- 认证、品牌要求和安全设计边界更严格

## 7. 从 EMV 内核角度看四类形态差异

| 维度 | 传统 POS | mPOS | SmartPOS | SoftPOS |
| --- | --- | --- | --- | --- |
| Kernel 部署 | 本机 | 外设或配套方案 | 本机 | 本机或平台方案 |
| 读卡链路 | 本机硬件 | 外设链路 | 本机硬件 | 手机 NFC |
| PIN 输入 | 专用 PIN Pad | 多在外设 | 设备安全输入 | 受场景与规则限制 |
| 日志定位 | 单机为主 | 多端联查 | 单机为主 | 手机侧为主 |
| 升级发布 | 终端包发布 | App + 外设双版本 | App / 固件协同 | App 远程发布 |
| 参数治理 | 传统 TMS | TMS + 设备侧配置 | TMS + App 配置 | 云端治理更强 |

## 8. 安全边界差异

如果只从功能看，四类设备都能“跑支付”；但从安全边界看，差异非常大。

### 8.1 传统 POS

- 专用设备
- 安全域边界清晰
- 适合传统密钥管理和外设隔离模型

### 8.2 mPOS

- 安全能力更多依赖外设
- 手机侧业务 App 与支付外设之间的信任链必须清晰

### 8.3 SmartPOS

- 支付能力和通用 Android 能力在同设备共存
- 更需要关注应用隔离、权限、系统完整性和厂商安全组件

### 8.4 SoftPOS

- 最强调设备可信、运行环境可信、远程风控可信
- 需要把“终端安全”理解成设备能力、系统状态、App 完整性、风控策略的组合问题

## 9. 从认证和测试角度看差异

### 9.1 传统 POS

- 认证路径成熟
- 更容易沿用既有经验

### 9.2 mPOS

- 不仅要看支付流程，还要看外设协同和移动端集成
- 现场问题容易分散到多个组件

### 9.3 SmartPOS

- 不同厂商平台差异大
- 需同时关注内核、外设 SDK、系统权限和业务 App 行为

### 9.4 SoftPOS

- 除了 EMV 本身，还要额外重视设备适配、安全检测、品牌与方案要求
- 很多问题不是纯内核问题，而是终端能力和产品策略问题

## 10. 架构设计建议

### 10.1 做统一抽象，别让业务层感知过多设备差异

建议把以下能力抽成统一接口：

- 读卡入口
- CVM / PIN 能力
- 内核结果
- 联机数据快照
- 设备能力探测

### 10.2 把设备能力和品牌规则分离

不要把“这是 SoftPOS，所以只能这样做”和“这是某品牌规则要求”写死在一起。

更好的方式是区分：

- 设备能力约束
- 品牌 / 方案规则约束
- 产品策略约束

### 10.3 对多形态方案采用 profile 化治理

建议至少维护：

- Device profile
- Kernel profile
- Host mapping profile
- Parameter profile

## 11. 日志与排错建议

终端形态越复杂，越不能只打一层日志。

### 传统 POS / SmartPOS

建议至少保留：

- 设备能力日志
- APDU / Kernel 阶段日志
- 联机前后快照
- 业务状态日志

### mPOS

建议至少保留：

- 外设连接日志
- 读卡器固件版本
- SDK 调用链日志
- 手机 App 状态日志

### SoftPOS

建议至少保留：

- NFC 会话日志
- 设备能力检测结果
- 风控 / 完整性检查结果
- 内核和联机快照

## 12. 怎么做产品选型

如果你的目标是快速做一个可大规模部署的轻量方案，可能更偏 SoftPOS。

如果你的目标是：

- 安全边界更传统
- 外设能力更完整
- 打印 / 外接模块需求更多

则可能更偏传统 POS 或 SmartPOS。

如果你的目标是手机为主、外设补足刷卡与 PIN 能力，则更偏 mPOS。

所以选型不要只问“能不能做 EMV”，而要同时问：

- 需要哪些支付路径
- 需要哪些外设能力
- 需要怎样的安全边界
- 需要怎样的运维和发布方式
- 认证和项目交付成本能否接受

## 13. 一个简单结论

四类终端都可以承载 EMV，但工程重点不同：

- 传统 POS：稳定、成熟、边界清晰
- mPOS：移动集成强，但多端协同复杂
- SmartPOS：业务扩展强，但平台差异大
- SoftPOS：部署轻、扩展快，但设备和安全治理要求更高
