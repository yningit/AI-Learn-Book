# 22 - 非接触内核与 Entry Point 深化专题

## 1. 为什么这一册值得单独展开

很多团队在接触式 EMV 跑通之后，会以为把接口切到 NFC 就算完成了非接支持。

真正做项目时很快会发现，非接问题并不是“少几条 APDU”这么简单，而是会集中体现在：

- 多品牌 Entry Point 路由
- TTQ / Reader limit / CVM limit 配置
- Outcome 抽象和 UI 指示
- Tap、移卡、重复感应、超时等用户交互行为
- 非接品牌差异和终端能力约束

所以这一册的目标，是把“非接为什么总让人觉得更绕”讲透。

## 2. 非接交易和接触交易最大的不同是什么

接触交易更像：

- 已经明确连上一张卡
- 终端和卡在较稳定的物理链路上执行完整流程

非接交易更像：

- 先在更短、更不稳定的交互窗口内识别候选应用
- 再快速决定是否由某个内核接管
- 再把结果抽象成统一 Outcome 给上层 UI 和收银逻辑

所以非接的难点通常不只是 Kernel 本身，而是：

- 路由
- 时间窗口
- 终端能力表达
- 用户交互收口

## 3. Entry Point 到底在解决什么问题

可以把 Entry Point 理解成“非接交易的总调度器”。

它通常负责：

- 接收卡片感应事件
- 收集候选应用或候选 kernel 信息
- 结合终端能力、品牌规则、参数配置做路由
- 把交易交给某个 contactless kernel
- 最后把内核结果抽象成统一 Outcome

## 4. 为什么非接场景特别依赖 Entry Point

因为非接经常同时具备下面几个特点：

1. 候选应用可能不止一个
2. 多品牌规则在入口阶段就开始分化
3. 用户交互时间短，不能把太多复杂性暴露给上层
4. 终端最终要输出统一 UI，而不是让上层自己理解每个品牌专有结果

## 5. 一个推荐的非接架构分层

建议至少区分下面几层：

### 5.1 Reader / NFC 控制层

- 卡检测
- 防冲突 / 激活
- 链路时序控制
- 场强、移卡、重试等底层事件

### 5.2 Entry Point 层

- 候选发现
- 路由决策
- Kernel 选择
- 统一 Outcome 组装

### 5.3 Scheme Kernel 层

- 执行品牌定义的 contactless kernel 流程
- 输出品牌侧交易结果与中间对象

### 5.4 POS App / UI 层

- 展示提示语
- 处理用户重试、取消、回退
- 处理联机与业务收口

## 6. TTQ 为什么在非接里这么重要

`9F66` TTQ（Terminal Transaction Qualifiers）可以理解为：

- 终端向卡片声明“我具备哪些非接能力、偏好和约束”

工程上它经常影响：

- 是否支持联机
- 是否支持脱机数据认证
- 是否支持联机 PIN / 签名 / CVM 能力
- 是否支持特定 reader 行为

### 6.1 初学者最容易误解的地方

不要把 TTQ 当成：

- 一个可随便固定写死的常量

更好的理解是：

- 它是终端能力、产品策略、品牌规则和参数配置共同作用下的结果

### 6.2 工程建议

- TTQ 不要散落在代码里拼接
- 应由参数 + 设备能力 + 品牌 profile 共同生成
- 日志里要打印最终 TTQ 和生成依据

## 7. Reader limit / CVM limit / Floor limit 为什么总容易混

非接常见的“限额”经常被混成一团，但它们解决的问题并不一样。

### 7.1 Reader Contactless Limit

- 终端愿意受理该非接交易的金额边界

### 7.2 CVM Limit

- 超过某金额后是否需要持卡人验证

### 7.3 Floor Limit

- 更多偏向是否倾向联机或进入风险控制考量

### 7.4 工程上为什么容易出错

- 参数名字相似
- 接触与非接共用了一套错误抽象
- 不同品牌对这些限制的使用方式不完全相同

## 8. Outcome 为什么是非接的核心抽象

在非接里，业务层通常不应该直接理解大量内核专有细节，而应该消费一个统一的 Outcome。

一个好的 Outcome 至少应包含：

- 最终结果：批准 / 拒绝 / 尝试其他接口 / 需要再次感应 / 终止
- UI 指示：显示什么、是否蜂鸣、是否点亮指示灯
- 是否允许重试
- 是否需要改走接触
- 是否需要联机、签名、打印等后续动作

### 8.1 为什么 Outcome 做不好会出大问题

- UI 和真实交易状态不一致
- 不同品牌返回值泄漏到业务层
- 现场排错时无法判断错误发生在入口、内核还是业务层

## 9. 非接最常见的 6 类问题

### 9.1 能感应到卡，但始终进不到正确 Kernel

优先看：

- 候选应用识别
- Kernel ID / route profile
- AID 映射
- Entry Point 决策日志

### 9.2 小额免密行为不符合预期

优先看：

- TTQ
- CVM limit
- Reader 配置
- 品牌 profile

### 9.3 UI 提示和交易真实状态不一致

优先看：

- Outcome 组装层
- UI 映射表
- 业务层是否覆盖内核结果

### 9.4 同一张卡偶发成功、偶发失败

优先看：

- 场强和移动速度
- 重复激活
- 会话超时
- 终端 NFC 控制器行为

### 9.5 提示改插卡，但其实应允许重试

优先看：

- Outcome 中的 retry / alternate interface 标志
- UI 规则是否过度简化

### 9.6 不同品牌在同一终端表现差异大

优先看：

- 非接参数 profile 是否独立
- 品牌 Entry Point 路由规则
- Outcome 和限额是否被统一逻辑错误覆盖

## 10. 一个推荐的 Entry Point 日志结构

为了让现场可查，建议 Entry Point 层至少打印：

- Card detected
- Candidate list
- Route decision
- Selected kernel
- TTQ / 关键 reader 参数
- Final outcome
- Retry / alternate interface 建议

这样问题一出来，团队能快速知道：

- 是没识别到候选
- 是路由错了
- 是 kernel 执行错了
- 还是 Outcome 映射错了

## 11. 路由设计建议

### 11.1 让 Entry Point 只做路由，不做品牌核心逻辑

否则很容易变成：

- 入口层塞满品牌条件分支
- 任何品牌变更都影响公共入口

### 11.2 统一抽象这些对象

建议至少有：

- `KernelCandidate`
- `RouteContext`
- `RouteDecision`
- `KernelOutcome`
- `ReaderAction`

### 11.3 参数化而不是硬编码

尤其是：

- AID 与 kernel 的映射
- 限额
- TTQ 组合
- UI/outcome 映射

## 12. 非接和产品形态之间的关系

如果终端是：

- 传统 POS：非接能力多由专用读卡器控制
- SmartPOS：需要同时面对 Android 平台和厂商 SDK 差异
- SoftPOS：NFC 能力、设备碎片化和系统约束会进一步放大非接问题

所以同样的 Entry Point 设计，在不同终端形态上的稳定性要求也不同。

## 13. 一套实用的非接排查顺序

遇到非接异常时，建议先按下面顺序排：

1. 卡是否被稳定检测和激活
2. 候选应用是否识别正确
3. Entry Point 是否路由到正确 kernel
4. TTQ 和 reader 参数是否符合预期
5. Kernel 是否给出合理结果
6. Outcome 是否被正确组装并传给 UI / 业务层

## 14. 这一册读完后应达到什么程度

读完后，你应该能：

- 理解 Entry Point 不是“非接的前几步”，而是独立架构层
- 区分 TTQ、限额、Outcome、路由各自的工程职责
- 知道非接问题为什么经常落在路由和 UI 收口，而不只是 APDU 本身
