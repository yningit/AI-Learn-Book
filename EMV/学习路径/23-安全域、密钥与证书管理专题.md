# 23 - 安全域、密钥与证书管理专题

## 1. 为什么安全与密钥管理必须单独成册

EMV 很容易被理解成“APDU + 流程控制”，但真正让它成为支付系统的，是背后的安全信任体系。

项目里很多最难排的问题，其实都和下面这些边界有关：

- 哪些对象是终端参数，哪些是密钥材料
- 哪些密钥用于 ODA，哪些用于联机授权
- 哪些能力在终端本地，哪些应由 SAM / HSM / Host 持有
- 测试环境和生产环境怎样严格隔离

这一册的目标不是展开规范级算法证明，而是讲清楚工程边界和治理方法。

## 2. 先把常见“安全对象”分层

建议把 EMV 里的安全相关对象分成 4 层去理解：

### 2.1 信任根层

- CAPK
- RID
- CA Index

它们主要解决：

- 终端如何信任卡片证书链

### 2.2 卡片 / 发卡行安全对象层

- Issuer Public Key
- ICC Public Key
- Signed Static / Dynamic Application Data
- IAD / CVR 等发卡行相关数据

它们主要解决：

- 卡片真实性、动态认证和发卡行上下文

### 2.3 终端与联机安全层

- UN
- ARQC / ARPC 相关参与数据
- 主机 HSM 中的校验能力

它们主要解决：

- 联机授权请求和响应的安全校验

### 2.4 终端设备安全层

- PIN 输入安全域
- SAM / SE / TEE / 安全芯片
- 设备完整性和密钥注入控制

它们主要解决：

- 终端本身是否可信，以及敏感能力放在哪里

## 3. CAPK 到底是什么，不是什么

CAPK 是终端本地预置的 CA 公钥。

它是：

- ODA 信任链的起点
- 参数治理对象
- 需要版本化和环境隔离的内容

它不是：

- 卡片公钥
- 每笔交易动态生成的值
- 主机授权密钥本身

## 4. 为什么 CAPK 治理经常决定 ODA 成败

如果以下任一项出错，ODA 往往就会失败：

- RID 命中错误
- CA Index 错误
- CAPK 缺失
- CAPK 测试 / 生产混用
- CAPK 已过期或版本旧

所以 ODA 问题往往不是纯算法错误，而是：

- 参数管理错误
- 环境隔离错误
- 版本治理错误

## 5. ARQC / ARPC 相关密钥边界应该怎么理解

一个非常重要的工程原则是：

- 终端不应把“主机验证 ARQC 的密钥能力”当成本地普通业务能力来实现

更直白一点说：

- 终端负责采集、组织和上送 ARQC 相关数据
- 主机 / HSM 负责真正的验证与响应计算

### 5.1 终端侧重点

- 正确收集 `9F26`、`9F27`、`9F10`、`9F36`、`9F37`、`95` 等对象
- 正确构造 Field 55
- 正确消费 ARPC 与 Issuer Script

### 5.2 主机侧重点

- 在受控安全环境中验证 ARQC
- 计算 ARPC
- 管理联机密钥材料和轮换

## 6. SAM、HSM、TEE、SE 应该怎样区分

这些概念经常被混用，但工程上最好分清：

### 6.1 SAM

- 更偏终端侧的安全模块
- 常用于某些本地安全能力或密钥封装场景

### 6.2 HSM

- 更偏主机侧或后台侧安全硬件
- 常用于密钥管理、验签、授权相关安全计算

### 6.3 TEE / SE / 安全芯片

- 更偏设备级安全隔离能力
- 在 SmartPOS / SoftPOS 等形态里尤其重要

## 7. PIN 和普通 EMV 数据的安全边界不同

初学者容易把“EMV 数据”与“PIN 安全”混在一起。

更好的理解是：

- AIP、AFL、TVR、TSI、CID 等更偏交易控制对象
- PIN 输入、PIN Block、PIN 安全通道属于另一层更敏感的安全域问题

工程上应避免：

- 在普通日志中记录敏感 PIN 相关内容
- 在普通应用层暴露不必要的敏感接口

## 8. 测试环境和生产环境为什么必须严隔离

这在文档里看似老生常谈，但项目里最容易踩坑。

至少要隔离：

- CAPK
- 主机地址与证书
- 密钥版本
- 参数包
- 白名单 / 路由 / 风控开关

### 8.1 一类经典事故

- 测试环境能跑
- 生产环境突然 ODA 或联机失败
- 代码没变，真正变化的是环境与密钥边界

## 9. 密钥生命周期应该怎么治理

建议把安全对象都纳入清晰的生命周期管理：

1. 申请 / 审批
2. 注入 / 下发
3. 生效确认
4. 版本记录
5. 轮换 / 失效
6. 回滚或作废

### 9.1 为什么工程团队也要关心生命周期

因为很多“偶发问题”其实是：

- 版本切换时未同步
- 轮换后有终端没更新
- 日志里根本看不到当前生效版本

## 10. 一个建议的安全日志边界

日志必须能帮助排错，但不能泄露敏感数据。

建议：

- 打印 CAPK 标识和版本，不打印完整敏感材料
- 打印命中的 RID / Index，不打印不必要原始密钥内容
- 打印 ARQC / ARPC 相关标签是否存在、长度是否正确
- 对极敏感字段做脱敏或摘要处理

## 11. 常见安全治理问题

### 11.1 CAPK 配置散落在多个模块

风险：

- 版本不一致
- ODA 排查困难

### 11.2 测试 / 生产材料混用

风险：

- 环境结论失真
- 上线事故

### 11.3 设备完整性与支付能力耦合不清

风险：

- 尤其在 SmartPOS / SoftPOS 场景下，难以确定哪些设备状态会影响支付可信性

### 11.4 日志完全看不到安全版本信息

风险：

- 问题发生后无法确认终端到底加载了哪套材料

## 12. 工程实现建议

### 12.1 安全对象统一仓库化管理

建议把下面对象统一纳入版本仓库：

- AID 参数
- CAPK
- 风险参数
- 终端能力参数
- 主机映射 profile

### 12.2 对环境做强约束

至少应支持：

- 测试 / 生产显式标识
- 版本可查询
- 下发记录可审计

### 12.3 把“安全边界”写进架构，而不只写进文档

比如：

- 普通业务模块拿不到敏感接口
- Host Adapter 只能消费必要字段
- 安全相关配置只能通过受控通道更新

## 13. 一套安全排查顺序

遇到 ODA、联机验签或环境差异问题时，建议按下面顺序查：

1. 当前环境是什么
2. 当前参数 / CAPK / profile 版本是什么
3. 是否命中了正确 RID / Index / AID profile
4. 主机与终端是否使用了同一套环境配置
5. 关键安全对象是否被错误编码、截断或替换

## 14. 这一册读完后应达到什么程度

读完后，你应该能：

- 区分 ODA 信任链和联机授权密钥边界
- 知道 CAPK、SAM、HSM、TEE 各自大致落在哪一层
- 理解很多“安全问题”本质上是治理和边界问题，而不只是密码学实现问题
