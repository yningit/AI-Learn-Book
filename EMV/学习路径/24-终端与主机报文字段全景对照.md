# 24 - 终端与主机报文字段全景对照

## 1. 为什么这一册很重要

很多 EMV 项目到了联机阶段才暴露出一个事实：

- 内核对象你看懂了
- 主机报文字段你也知道一些
- 但两者之间到底怎么映射，却没有一份清晰的“总对照表”

这会直接导致：

- Field 55 缺标签或标签来源不清
- 金额、日期、交易类型、终端能力在主机侧解释不一致
- 日志和主机报文没法一一对上

这一册的目标，就是把“终端里发生的事”和“主机上看到的字段”放到同一个坐标系里。

## 2. 先建立一个总心智模型

从终端到主机，大致会经历下面几层转换：

```text
Card / Kernel Objects
  -> EMV Context
	  -> Host Mapping Layer
		  -> ISO8583 Fields
			  -> Host Risk / Auth Processing
```

最容易出问题的，通常是中间这两层：

- `EMV Context -> Host Mapping`
- `Host Mapping -> ISO8583`

## 3. 哪些字段最值得优先对齐

### 3.1 基础业务字段

- 交易金额
- 币种
- 交易日期
- 交易类型
- 终端流水号
- 批次号 / 参考号

### 3.2 核心 EMV 字段

- `82` AIP
- `95` TVR
- `9A` Transaction Date
- `9C` Transaction Type
- `9F02` Amount, Authorised
- `9F10` IAD
- `9F1A` Terminal Country Code
- `9F26` Application Cryptogram
- `9F27` CID
- `9F34` CVM Results
- `9F36` ATC
- `9F37` UN

### 3.3 联机后处理字段

- ARC / Response Code
- Authorization Code
- ARPC
- Issuer Script

## 4. 一个推荐的映射分层

### 4.1 Kernel 负责

- 提供可信的 EMV 交易对象
- 不直接关心某个主机字段编号怎么组装

### 4.2 Host Mapping 层负责

- 决定哪些对象要上送
- 决定放入 Field 55 还是其他 ISO8583 字段
- 处理品牌 / 主机差异

### 4.3 ISO8583 Adapter 负责

- 把结构化对象编码成具体报文
- 处理长度、BCD、LLVAR / LLLVAR 等技术细节

这样做的好处是：

- 内核不被主机差异污染
- 字段问题能快速定位在 mapping 层，而不是误判成 kernel bug

## 5. 一张最实用的终端到主机对照表

| 终端/内核对象 | 常见 Tag / 来源 | 主机侧常见去向 | 排查重点 |
| --- | --- | --- | --- |
| 金额 | `9F02` / POS 输入 | 金额字段 + Field 55 | BCD 编码、单位、退货/消费类型 |
| 交易类型 | `9C` | 交易类型字段 + Field 55 | 消费、撤销、预授权等映射 |
| 日期 | `9A` | 交易日期字段 + Field 55 | 本地时间 vs 主机时区 |
| AIP | `82` | Field 55 | 来源是否为 GPO 结果 |
| TVR | `95` | Field 55 / 主机风控输入 | 位图是否按真实事实置位 |
| IAD | `9F10` | Field 55 / 发卡行风控输入 | 截断、编码、品牌私有解释 |
| AC | `9F26` | Field 55 / 主机验签 | 长度、二进制传输 |
| CID | `9F27` | Field 55 | ARQC / TC / AAC 解释 |
| CVM 结果 | `9F34` | Field 55 / 风控 | 是否与 UI/设备实际一致 |
| ATC | `9F36` | Field 55 | 递增是否异常 |
| UN | `9F37` | Field 55 / 主机验签输入 | 随机质量与重复问题 |

## 6. Field 55 最容易踩的坑

### 6.1 以为“有 Field 55 就够了”

不对。真正要问的是：

- Tag 是否齐全
- 值是否来自正确来源
- 编码是否正确
- 主机是否按同样语义解释

### 6.2 把文本和字节流混用

高频问题包括：

- 十六进制串被当成 ASCII
- 长度按字符数而不是字节数计算
- 中间层转义或裁剪数据

### 6.3 把不同主机环境的要求混在一起

建议始终区分：

- Scheme mapping
- Host mapping
- Environment mapping

## 7. 一张联机排查速查表

| 现象 | 优先检查对象 | 最常见责任边界 |
| --- | --- | --- |
| ARQC 校验失败 | `9F26`、`9F10`、`9F36`、`9F37`、`95` | Host mapping / 主机环境 |
| 主机说缺少 EMV 数据 | Field 55 标签集合 | ISO8583 adapter / Host mapping |
| 主机批准但终端失败 | ARPC、Script、AC2 | 联机后处理 |
| 主机未落账但终端显示成功 | 授权状态、参考号、持久化状态 | POS App / Host adapter |
| 不同环境行为不一致 | 参数版本、CAPK、主机开关 | 环境配置治理 |

## 8. 不同品牌为什么要拆独立 mapping

因为品牌差异常常体现在：

- IAD / CVR 解释
- 必传标签集合
- 特定 Tag 的使用方式
- 联机后返回值消费方式

如果把所有品牌都揉进一套硬编码 mapping，最终会导致：

- switch 越来越多
- 联调问题难以定位
- 某品牌修改影响其他品牌

## 9. 一个推荐的数据结构

建议在工程里至少沉淀下面几种结构化对象：

```text
EmvContext
HostMappingProfile
Field55Snapshot
Iso8583RequestView
Iso8583ResponseView
OnlineProcessingResult
```

其中最关键的是：

- `Field55Snapshot` 要能直接回看联机前的上送内容
- `Iso8583RequestView` 要能展示主机真正收到的结果

## 10. 日志应该怎么打才最有用

建议至少能同时看到：

1. 内核原始对象快照
2. Field 55 序列化前结构
3. ISO8583 请求关键字段
4. 主机响应字段
5. 联机后处理结果

这样现场排查时能快速回答：

- 是卡片对象错了
- 还是 mapping 错了
- 还是主机解释错了

## 11. 这一册读完后应达到什么程度

读完后，你应该能：

- 把内核对象和主机字段放到一张图里理解
- 知道 Field 55 问题为什么大多不是“少个 Tag”这么简单
- 知道主机联调问题最适合从 mapping 层切入
