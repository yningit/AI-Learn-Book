# 25 - 认证资料、测试包与实验室送测专题

## 1. 为什么这一册必须补上

很多团队做到这里时，代码已经能跑，日志也能看，但一到送测和认证阶段就开始混乱：

- 不知道要给实验室准备哪些资料
- 不知道测试包、参数包、版本包怎么冻结
- 不知道开发自测、内部 QA、品牌测试、实验室送测分别是什么关系

所以这一册专门解决“从工程实现走向认证交付”这件事。

## 2. 先理解几类常见测试/认证活动

### 2.1 开发自测

- 验证功能能跑通
- 验证日志和基本对象正确

### 2.2 内部 QA / 集成测试

- 验证业务集成
- 验证参数、主机、设备组合

### 2.3 Kernel L2 / 品牌测试

- 验证内核行为是否符合规范和品牌要求

### 2.4 实验室送测 / 认证准备

- 验证提交材料是否完整
- 验证测试环境与资料一致
- 验证问题闭环是否可追踪

## 3. 实验室最关心的不是“你说过了”，而是“你能证明”

所以送测前必须准备的不只是程序包，还包括：

- 版本信息
- 参数信息
- 环境信息
- 支持能力清单
- 已知限制说明
- 问题回归记录

## 4. 一套建议的送测资料包结构

建议按下面结构整理：

### 4.1 版本资料

- 软件版本号
- 固件版本号（如适用）
- 外设版本号（读卡器、PIN pad、NFC 组件等）
- 参数版本号

### 4.2 能力说明

- 支持的交易类型
- 支持的接口类型：接触 / 非接
- 支持的品牌 / 组织
- 支持的 CVM 路径
- 支持的联机 / 脱机范围

### 4.3 配置资料

- AID 列表
- CAPK 列表
- 风险参数清单
- 非接参数和限额清单
- Host mapping 或联机字段说明

### 4.4 测试资料

- 自测通过清单
- 已执行 case 列表
- 已知问题清单
- 日志样例和采集方式

## 5. 送测前最容易遗漏的 10 个项目

1. 没冻结参数版本
2. 没区分测试 / 生产材料
3. 没记录读卡器或外设版本
4. 没准备失败 case 的日志
5. 不清楚某些功能是否“支持但有限制”
6. 没有统一的问题单编号
7. 不知道实验室拿到的是哪一版包
8. Host mapping 文档不完整
9. 非接参数和接触参数混在一起
10. 送测样机与开发样机配置不一致

## 6. 一份最实用的送测检查清单

| 类别 | 必查项 |
| --- | --- |
| 版本 | App、内核、固件、参数、CAPK 版本是否可追踪 |
| 设备 | 样机型号、硬件版本、外设版本是否明确 |
| 能力 | 支持品牌、接口、交易类型、CVM 是否明确 |
| 配置 | AID、CAPK、风险参数、非接参数是否冻结 |
| 测试 | 自测结果、失败回归、已知问题是否整理 |
| 联调 | 主机环境、字段说明、超时/冲正策略是否明确 |
| 日志 | 日志采集方式、字段说明、问题回放方式是否明确 |

## 7. 如何准备测试包和参数包

建议把“程序包”和“参数包”分开管理，但在送测时一起冻结：

- 程序包：代码与固件
- 参数包：AID、CAPK、限额、TAC、reader 参数等

送测时最好同时给出：

- 程序包版本
- 参数包版本
- 两者对应关系

否则实验室发现问题时，你可能根本不知道测的是哪一组组合。

## 8. 一套建议的问题闭环方式

每个实验室问题建议至少记录：

- case 名称 / 编号
- 失败现象
- 日志位置
- 根因分类：代码 / 参数 / 环境 / 主机 / 设备
- 修复版本
- 回归结论

这样后面做复测时，团队不会反复丢上下文。

## 9. 送测资料中最该提前准备的 3 类说明

### 9.1 支持范围说明

明确写清：

- 什么支持
- 什么不支持
- 什么条件下支持

### 9.2 已知限制说明

这比“全部支持”更真实，也更能避免沟通误差。

### 9.3 日志说明

告诉实验室：

- 关键日志在哪里
- 如何导出
- 哪些字段最重要

## 10. 送测前推荐的内部彩排顺序

1. 开发自测
2. 内部 QA 回归
3. 参数和环境冻结
4. 样机一致性检查
5. 送测资料包审阅
6. 按实验室使用视角做一次模拟执行

## 11. 这一册读完后应达到什么程度

读完后，你应该能：

- 理解“代码通过”和“送测准备充分”是两回事
- 知道送测真正需要准备哪些资料与版本信息
- 建立测试包、参数包、日志、问题单之间的闭环意识
