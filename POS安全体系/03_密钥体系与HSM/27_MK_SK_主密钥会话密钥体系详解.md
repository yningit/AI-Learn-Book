# MK/SK 主密钥会话密钥体系详解

## 1. 为什么已经有 DUKPT，还要学 MK/SK

很多学习者会把真实世界想得过于整齐，仿佛：

- 新一点的终端都用 DUKPT。
- 不用 DUKPT 的就只是旧系统。

这不够准确。

现实里经常同时存在：

- DUKPT 体系
- MK/SK 体系
- 用途键分离体系
- 厂商自定义的主键 / 工作键派生体系

所以如果你要理解 POS 的整体安全，而不是只理解某一个方案，就必须看懂 `MK/SK`。

## 2. MK/SK 是什么

`MK/SK` 一般可理解为：

- `Master Key / Session Key`，即主密钥 / 会话密钥体系。

最核心的思路是：

- 先在终端和主机之间建立一个受控主密钥关系。
- 再基于它装载、协商、保护或更新会话密钥。
- 具体交易或一段通信期间，用会话密钥处理数据加密、MAC 或 PIN 相关操作。

如果你把 DUKPT 理解成“每笔换锁”，那么 MK/SK 更像：

- “先约定主锁，再定期换本次会话的子锁。”

## 3. MK/SK 体系里常见会看到什么名词

不同厂商命名不完全一致，但常见语境包括：

- TMK：终端主键或终端主控层材料
- TPK：终端 PIN 工作键
- TAK：终端 MAC / 认证工作键
- TDK：终端数据加密工作键
- Session Key：本次会话的工作键
- Key Set / Key Slot：终端内部的逻辑密钥集合

这里要抓住的不是缩写本身，而是三层关系：

1. 高层主控键
2. 中间分发或保护键
3. 具体用途工作键

## 4. MK/SK 的基本工作方式

一个简化模型通常是：

1. 终端持有某个主控键或受主控键保护的工作键集合。
2. 主机侧在 HSM 中保存对应密钥关系。
3. 终端开机、签到、会话建立或参数下载时，装载或激活本次 Session Key。
4. 在会话期间，用该 Session Key 或其用途变体处理 MAC、Data、PIN 等操作。
5. 会话结束、超时、重签或轮换时，再更新下一把 Session Key。

因此 MK/SK 的轮换粒度通常更接近：

- 一次签到
- 一次会话
- 一段受控交易周期

而不是像 DUKPT 那样天然做到：

- 每一笔交易都从派生体系得到不同工作键。

## 5. MK/SK 与 DUKPT 的核心区别

### 5.1 密钥变化粒度不同

- `DUKPT` 更偏每笔交易变化。
- `MK/SK` 更偏会话或周期变化。

### 5.2 状态依赖点不同

- `DUKPT` 强依赖 KSN 和派生状态。
- `MK/SK` 强依赖终端签到、会话建立、密钥装载和同步状态。

### 5.3 主机处理模式不同

- `DUKPT` 主机常通过 KSN 重建交易密钥。
- `MK/SK` 主机常通过终端身份、会话上下文、键组配置来选取对应工作键。

### 5.4 运维重点不同

- `DUKPT` 更容易把风险聚焦在派生域、KSN 和 BDK 管理。
- `MK/SK` 更容易把风险聚焦在签到流程、工作键同步、轮换、过期和错配。

## 6. MK/SK 什么时候常见

现实里常见于：

- 传统 POS 终端到主机通信体系
- 参数下载、管理报文、签到报文保护
- 某些终端厂商的静态或半动态工作键模型
- 历史系统和新系统并存的过渡架构

特别要注意的是：

- 就算主交易数据用了 DUKPT，终端管理、签到或某些控制报文仍可能用 MK/SK 或固定用途键语境处理。

## 7. MK/SK 为什么没有过时

它之所以仍大量存在，不是因为大家都没升级，而是因为它在很多场景有明确工程价值：

- 模型清晰，易于和传统终端管理流程结合。
- 便于按设备、按会话、按用途做运营控制。
- 对某些管理报文和历史主机协议兼容性更好。
- 与 HSM 的传统终端密钥命令体系衔接成熟。

这并不意味着它一定优于 DUKPT，而是：

- 两者针对的历史背景、运维模式和接入生态并不完全相同。

## 8. MK/SK 体系的常见风险点

### 8.1 会话状态不同步

例如：

- 终端以为已经切到新 Session Key。
- 主机仍按旧 Session Key 校验。

这会表现为：

- MAC 校验失败
- 数据解密失败
- 签到后首笔报文失败

### 8.2 工作键装载路径不清

如果没有明确回答：

- 工作键由谁生成
- 用哪把主控键包装
- 在哪里解包导入

那系统很容易变成“能跑通，但边界很模糊”。

### 8.3 多环境串线

MK/SK 很依赖键组和终端身份映射，一旦：

- 生产终端指向测试主机
- 测试终端拿到生产键组

就会出现既危险又难排查的问题。

### 8.4 用途不分离

即使采用 MK/SK，也不代表可以把：

- Data
- PIN
- MAC

混用同一把工作键。

真正成熟的体系仍应强调：

- 按用途分离
- 按环境分离
- 按设备或商户域分离

## 9. 如何判断一个项目里是 MK/SK 主路径还是 DUKPT 主路径

你可以直接看下面几个信号：

1. 主机是否依赖 `KSN` 来恢复交易密钥。
2. 终端是否有明显的签到换会话键流程。
3. 报文里是否出现 terminal key set、session key index、sign-in key update 之类语义。
4. HSM 调用更偏 DUKPT 派生命令，还是更偏终端工作键导入、解包、MAC、PIN 命令。

很多真实系统不是二选一，而是：

- 交易数据一条路径
- 管理报文另一条路径

## 10. 学习 MK/SK 时最重要的结论

不要把它理解成“落后版 DUKPT”。

更准确的理解是：

- 它是另一类主机终端密钥运营模型。

只有把 MK/SK、DUKPT、用途键分离、HSM 层级放到一起看，你才会真正看懂为什么一个 POS 项目里会同时出现 `TMK / TPK / TAK / BDK / IPEK / MAC Key / Data Key` 这些名词而不矛盾。
