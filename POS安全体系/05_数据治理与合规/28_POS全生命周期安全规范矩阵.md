# POS 全生命周期安全规范矩阵

## 1. 为什么需要一张规范矩阵

当文档只讲 DUKPT 或某一条链路时，很多人容易形成一种错觉：

- 只要交易数据加密做对，POS 安全就算完成了。

但真实项目里，控制点分散在：

- 设备设计
- 终端生产
- 安全注入
- 物流与安装
- 交易运行
- 参数和固件管理
- HSM 与密钥运营
- 日志与监控
- 返修和报废

而不同规范其实分别约束不同环节。

这篇的目标，就是把“技术主题”和“生命周期环节”放到同一张图里。

## 2. 先建立一个环节视图

可以把 POS 全生命周期粗分为九个阶段：

1. 方案设计与选型
2. 终端生产与供应链控制
3. 密钥生成、运输与注入
4. 发运、仓储、安装与激活
5. 交易运行与报文保护
6. 主机处理与解密域控制
7. 业务留存、去敏与令牌化
8. 运维、升级、返修与替换
9. 退役、报废、审计与事件响应

如果你只盯住第 5 阶段，就会漏掉大量更早或更后的风险。

## 3. 常见规范各自主要管哪一段

### 3.1 PCI DSS

更偏整体卡数据环境控制，包括：

- 网络分段
- 访问控制
- 日志审计
- 漏洞管理
- 变更管理
- 安全运营

它对第 4 到第 9 阶段都有影响，尤其是：

- 哪些系统接触卡数据
- 哪些环境需要被纳入审计范围

### 3.2 PCI PTS

更偏终端设备和 PIN 输入设备本身的安全能力，尤其影响：

- 第 2 阶段终端生产与选型
- 第 4 阶段部署使用
- 第 8 阶段返修与升级控制

### 3.3 PCI P2PE

更偏：

- 读取点到解密点的整体保护方案和管理闭环

它强关联：

- 终端
- 注入
- 传输
- 解密域
- 管理流程

### 3.4 PCI PIN

更偏：

- PIN 安全链路
- PIN Key 管理
- HSM PIN 域处理
- 双人控制与职责分离

### 3.5 PCI SSF

更偏支付软件本身的安全开发与发布控制，适用于：

- POS 应用
- 支付中间件
- 主机侧支付软件

它帮助你回答：

- 即便设备硬件安全，软件发布和维护是否仍会把风险重新带进来。

### 3.6 ANSI X9.24 / TR-31 / ISO 9564 / EMV 相关语境

这些内容更多提供工程化细则，例如：

- DUKPT、密钥管理、PIN 处理、key block、终端交互与卡片流程

它们往往不是替代 PCI，而是补充“怎么做得更工程化”。

## 4. 用生命周期来看，每一段到底该关注什么

### 4.1 方案设计与选型

要回答：

- 终端是否具备受控安全能力。
- 采用 DUKPT 还是 MK/SK，或者两者并存。
- PIN、Data、MAC 是否分离。
- 是否需要 tokenization。
- 商户侧哪些系统绝不能接触明文。

主要规范语境：

- PCI DSS
- PCI PTS
- PCI P2PE
- PCI PIN
- PCI SSF

输出物应包括：

- 架构图
- 数据流图
- 密钥层级图
- 环境边界图

### 4.2 终端生产与供应链控制

要回答：

- 设备型号和固件版本是否受控。
- 设备序列号和身份是否可追溯。
- 调试口和维护口是否关闭或受控。
- 出厂前后有没有被替换或混料风险。

主要规范语境：

- PCI PTS
- SRED 相关设备安全要求

输出物应包括：

- 设备台账
- 固件基线
- 供应链追溯记录

### 4.3 密钥生成、运输与注入

要回答：

- BDK、TMK、工作键由谁生成。
- 用什么方式安全运输。
- 是否使用 TR-31 / Key Block。
- 注入后如何校验、留痕、绑定设备。

主要规范语境：

- PCI PIN
- PCI P2PE
- ANSI X9.24
- TR-31

输出物应包括：

- 注入台账
- KCV 记录
- 双人控制记录
- 设备与密钥映射关系

### 4.4 发运、仓储、安装与激活

要回答：

- 设备送到谁手里。
- 是否部署到正确门店和环境。
- 激活是否拿到正确参数和证书。
- 现场替换时是否能识别设备身份。

主要规范语境：

- PCI DSS
- PCI P2PE
- 运营与审计要求

输出物应包括：

- 门店绑定记录
- 激活记录
- 环境归属记录

### 4.5 交易运行与报文保护

要回答：

- 敏感读取是否在安全边界内完成。
- 数据链路是否采用 DUKPT 或 MK/SK 等受控机制。
- 报文是否具备 MAC 或等价完整性控制。
- POS、网关、中间件是否只处理最少必要对象。

主要规范语境：

- PCI P2PE
- PCI PIN
- ANSI X9.24
- EMV 相关语境

输出物应包括：

- 报文规范
- 字段分级表
- 密钥用途表

### 4.6 主机处理与解密域控制

要回答：

- 解密是否始终在 HSM 或受控域中完成。
- 主机如何按终端身份、KSN、键组选择正确路径。
- 解密后哪些系统能看到什么级别的明文。

主要规范语境：

- PCI DSS
- PCI PIN
- HSM 控制语境

输出物应包括：

- HSM 路由表
- 解密域边界图
- 权限矩阵

### 4.7 业务留存、去敏与令牌化

要回答：

- 哪些业务必须保留真实数据。
- 哪些流程可以只用 token。
- detokenization 如何审批和审计。
- 报表、客服、BI 是否只消费去敏对象。

主要规范语境：

- PCI DSS
- Tokenization 设计实践

输出物应包括：

- 数据保留策略
- token 域设计
- 展示脱敏规则

### 4.8 运维、升级、返修与替换

要回答：

- 固件升级如何认证和留痕。
- 篡改告警如何处理。
- 返修前是否擦除密钥。
- 换机、补机、撤机如何更新台账和密钥状态。

主要规范语境：

- PCI DSS
- PCI PTS
- PCI P2PE
- PCI PIN

输出物应包括：

- 升级记录
- 返修记录
- 吊销记录
- 告警处置记录

### 4.9 退役、报废、审计与事件响应

要回答：

- 终端报废时是否完成密钥清除和资产注销。
- 历史 token、映射关系、日志和备份是否按策略销毁或留存。
- 发生泄露、串线、篡改、误注入时是否能追溯完整链路。

主要规范语境：

- PCI DSS
- PCI P2PE
- PCI PIN
- 组织内部安全事件响应规范

输出物应包括：

- 报废证明
- 审计留痕
- 事件复盘报告

## 5. 如何用这张矩阵指导文档扩展

如果把当前文档集继续扩展成“POS 数据安全教材”，建议按下面四条主轴补齐：

1. `设备主轴`：SRED、PTS、固件与供应链安全。
2. `链路主轴`：P2PE、DUKPT、MK/SK、PIN、MAC、报文完整性。
3. `密钥主轴`：HSM、LMK/ZMK/TMK/BDK、TR-31、注入与用途控制。
4. `数据治理主轴`：Tokenization、去敏、数据保留、日志治理、权限审计。

这样你就不会把“POS 安全”误收缩成“终端加密”这一小块。

## 6. 一句总结

POS 数据安全不是一个算法问题，而是一张覆盖设备、密钥、报文、系统、运维和审计的生命周期控制矩阵。只有把规范要求映射到每个环节的责任、台账和技术控制上，体系才算真正闭环。
