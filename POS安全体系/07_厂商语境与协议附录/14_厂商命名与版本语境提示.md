# 厂商命名与版本语境提示

## 1. 为什么这一篇有必要

当你开始读厂商手册、终端协议文档或老项目资料时，经常会看到一些名字：

- OnGuard
- DUKPT 2007
- DUKPT 2009
- 某某 Secure Format
- 某某 Enhanced DUKPT
- 某某 Host Decrypt Mode

这些名字很容易让人误以为：

- 它们都是行业统一标准名词。

实际上，很多时候并不是这样。真实情况通常是：

- 有些是标准体系的版本语境。
- 有些是厂商包装名称。
- 有些是项目内部约定称呼。
- 有些是“算法 + 报文封装 + 设备实现”的混合命名。

所以这一篇的目的不是给这些名字下绝对定义，而是教你：

- 遇到这类词时，应该怎样拆解和判断。

## 2. 先记住一个原则：名字本身不够，必须追问语境

比如别人说：

- “我们这里用的是 OnGuard。”
- “这个终端走 DUKPT 2007。”
- “主机只支持 2009。”

如果只听名字，信息其实远远不够。你至少要继续追问四个维度：

1. 它指的是哪一层。
2. 它影响的是算法、用途键、报文格式，还是设备兼容。
3. 它是行业标准名，还是厂商名，还是项目简称。
4. 主机侧和终端侧分别如何识别它。

## 3. 你应该先把名词拆成三层

### 3.1 算法层

例如：

- 3DES DUKPT
- AES DUKPT

这一层回答的是：

- 密钥派生和数据处理大致属于哪类密码体系。

### 3.2 用途和处理层

例如：

- Data Encryption
- PIN Encryption
- MAC / Authentication
- Variant Key / Usage Attribute

这一层回答的是：

- 同一体系下，这把键或这个报文到底用于什么。

### 3.3 报文和产品层

例如：

- 某厂商定义的密文封装格式
- 某设备的 host interface mode
- 某项目里对一组参数组合的简称

这一层回答的是：

- 主机到底收到什么格式的字段。
- 终端会带哪些标识。
- 如何知道走哪条解密路径。

## 4. OnGuard 这类名字应该怎么理解

在很多项目语境里，像 OnGuard 这样的名称更像：

- 某家厂商或某条产品线对其终端加密输出/安全封装方案的命名

它未必单独等于：

- 一个完整行业标准
- 一个单独算法
- 一个单独密钥类型

它更可能混合了下面几件事：

- 使用了哪类终端加密能力
- 是否带 KSN
- 报文字段如何封装
- 主机侧应如何识别和解包
- 底层采用的是哪种 DUKPT 或用途键体系

所以如果文档里只写了 OnGuard，而没有继续说明：

- 算法类型
- KSN 组织方式
- 用途键类型
- 主机接口字段

那其实信息还不够完整。

## 5. DUKPT 2007 / DUKPT 2009 这类说法怎么对待

这类按年份表达的说法，在项目中通常说明：

- 文档作者在区分某一代实现、某一批规范语境、某一组兼容模式，或者某个厂商版本命名

但你不能只根据年份就推断出全部细节。更稳妥的做法是追问：

- 它底层属于 3DES 体系还是 AES 体系。
- 它影响的是 KSN 结构、派生方式，还是报文格式。
- 主机端所谓“支持 2007 / 2009”，具体支持的是哪类输入输出。

也就是说，这类年份标签本身更像“版本线索”，不是完整技术定义。

## 6. 遇到厂商命名时，最应该建立哪张映射表

当你碰到陌生命名时，最有效的办法不是先猜，而是建立一张映射表。最低限度建议包含下面这些列：

- 文档里的名称
- 终端厂商/型号
- 所属层级：算法 / 用途 / 报文 / 产品模式
- 是否带 KSN
- 是否属于 DUKPT
- 是 Data / PIN / MAC 哪类用途
- 主机侧怎么识别
- HSM 侧怎么处理
- 是否和某个旧版本兼容

很多项目里一旦把这张表做出来，原本非常混乱的术语就会立刻清晰很多。

## 7. 为什么主机端经常说“支持某某模式”

主机侧所谓“支持某某模式”，往往不是一句简单的话，它通常隐含的是：

- 能识别特定报文结构
- 能从特定字段抽取 KSN 或终端标识
- 知道应该路由到哪组 BDK 或哪种用途键链路
- 知道如何对密文载荷做对应解包和解密

所以在工程沟通里，不要只问：

- “你们支不支持 OnGuard / 2009？”

更应该问：

- “你们支持的输入报文长什么样，KSN 在哪里，底层是哪种 DUKPT，用于哪种数据对象，HSM 如何处理？”

## 8. 为什么项目里最怕“大家都懂这个简称”

支付项目中有一种很高风险的沟通方式：

- 大家都用简称，但没人继续展开。

例如：

- “这是老的 2007 格式。”
- “这个厂商用的是 OnGuard。”
- “这个模式和之前那个差不多。”

这种说法在同一团队里短期可能省事，但一到下面这些场景就会出问题：

- 新人接手
- 跨团队联调
- HSM 切换
- 厂商替换
- 终端升级
- 历史兼容改造

所以文档化时，最好始终把“简称 -> 技术展开”的映射写出来。

## 9. 如何把这些名字落回当前文档集的知识框架

你可以把这类名称回归到我们已经建立的四个问题里：

### 9.1 它属于哪一层

- P2PE 体系层
- DUKPT 算法层
- 用途键层
- 报文封装层
- 厂商产品层

### 9.2 它保护的是什么对象

- PIN
- Data
- MAC
- 管理报文

### 9.3 主机依靠什么识别

- KSN
- 终端标识
- 报文字段头
- 配置路由

### 9.4 它需要哪类 HSM / 主机能力

- DUKPT 重建
- 用途键变换
- PIN 域处理
- MAC 校验

## 10. 什么时候你必须向厂商继续要资料

当你只拿到一个名称，但无法回答下面任一问题时，就应该继续向厂商或上游要资料：

1. KSN 在哪里。
2. 终端输出的密文块格式是什么。
3. 底层是 3DES DUKPT 还是 AES DUKPT，或其他体系。
4. 用于哪类数据对象。
5. 主机应使用什么 HSM 处理路径。
6. 是否有兼容模式、旧版格式或切换条件。

## 11. 一句总结

像 OnGuard、DUKPT 2007、DUKPT 2009 这类词，最重要的不是先背名字，而是先判断它们到底在描述：

- 算法
- 用途
- 报文格式
- 厂商模式
- 项目内部简称

只要把语境拆清楚，这些名字就不会再显得神秘；如果语境不清，再熟悉的名词也可能让项目沟通和实现走偏。
